# Worldtech B2B CRM (React + Supabase)

เวอร์ชัน rebuild จาก Google Apps Script เดิม ให้ deploy บน Netlify ได้

## โครงสร้าง
- **Frontend:** React + Vite (ธีมสี Navy/Yellow เดิม)
- **Backend/Database:** Supabase (Postgres + Auth)
- **PDF ใบเสนอราคา:** สร้างผ่าน browser print dialog (บันทึกเป็น PDF ได้ทันที ไม่ต้องพึ่ง Google Docs)

## ขั้นตอนติดตั้ง

### 1. สร้าง Supabase Project
1. ไปที่ https://supabase.com สร้างโปรเจกต์ใหม่ (ฟรีได้)
2. เปิด **SQL Editor** → New query → คัดลอกทั้งหมดจาก `supabase/schema.sql` มาวาง → Run
   - ไฟล์นี้จะสร้างตารางทั้งหมด, Storage bucket `attachments` สำหรับเอกสารแนบ, และระบบสิทธิ์ Admin/Sale ให้อัตโนมัติ
   - ไฟล์รันซ้ำได้เสมอ (idempotent) ถ้าอัปเดตโปรเจกต์ในอนาคตให้รันไฟล์นี้ทับได้เลย
3. ไปที่ **Authentication > Users** → เพิ่มผู้ใช้งาน (อีเมล/รหัสผ่าน) สำหรับพนักงานที่จะเข้าระบบ
   - หรือเปิด **Authentication > Providers** ตั้งค่า sign-up ถ้าต้องการให้สมัครเองได้
   - ทุกบัญชีใหม่จะได้สิทธิ์ **"พนักงานขาย" (sale)** โดยอัตโนมัติ
4. ไปที่ **Project Settings > API** คัดลอกค่า:
   - `Project URL` → ใช้เป็น `VITE_SUPABASE_URL`
   - `anon public` key → ใช้เป็น `VITE_SUPABASE_ANON_KEY`
5. ตั้งให้ตัวเองเป็น **Admin คนแรก** — กลับไปที่ SQL Editor แล้วรัน:
   ```sql
   update profiles set role = 'admin' where email = 'you@company.com';
   ```
   (ต้อง login เข้าระบบอย่างน้อย 1 ครั้งก่อน เพื่อให้มีแถวใน `profiles` ให้อัปเดต) หลังจากนั้นเมนู "ผู้ใช้งาน" จะปรากฏใน Sidebar ให้จัดการสิทธิ์คนอื่นต่อได้จากในแอปเลย

### 2. ตั้งค่าโปรเจกต์ในเครื่อง
```bash
npm install
cp .env.example .env
# แก้ .env ใส่ VITE_SUPABASE_URL และ VITE_SUPABASE_ANON_KEY ของคุณ
npm run dev
```
เปิด http://localhost:5173 แล้ว login ด้วยบัญชีที่สร้างไว้ในขั้นตอนที่ 1.3

### 3. ตั้งค่าข้อมูลบริษัท (สำหรับหัวใบเสนอราคา PDF) — ไม่บังคับ
ใน Supabase SQL Editor รัน:
```sql
insert into settings (key, value) values
  ('COMPANY_NAME', 'บริษัทของคุณ จำกัด'),
  ('COMPANY_ADDRESS', 'ที่อยู่บริษัท'),
  ('COMPANY_TAX_ID', '0-0000-00000-00-0'),
  ('COMPANY_PHONE', '02-xxx-xxxx'),
  ('COMPANY_EMAIL', 'contact@yourcompany.com'),
  ('COMPANY_LINE', '@yourlineid')
on conflict (key) do update set value = excluded.value;
```
`COMPANY_TAX_ID` และ `COMPANY_LINE` ไม่บังคับ — ถ้าไม่ตั้งค่าไว้ใบเสนอราคาจะไม่แสดงบรรทัดนั้น

## Deploy ขึ้น Netlify

### วิธีที่ 1: ผ่านเว็บ Netlify (ง่ายที่สุด)
1. Push โค้ดทั้งโฟลเดอร์นี้ขึ้น GitHub repo
2. ที่ Netlify → **Add new site > Import an existing project** → เลือก repo
3. Build command: `npm run build`, Publish directory: `dist` (ตั้งไว้ให้แล้วใน `netlify.toml`)
4. ไปที่ **Site settings > Environment variables** เพิ่ม:
   - `VITE_SUPABASE_URL`
   - `VITE_SUPABASE_ANON_KEY`
   - `SUPABASE_SERVICE_ROLE_KEY` — คัดลอกจาก Supabase **Project Settings > API > service_role** (⚠️ **ห้ามใส่ prefix `VITE_`** ไม่งั้น Vite จะฝังคีย์นี้ไปกับโค้ดฝั่ง browser ซึ่งเท่ากับเปิดสิทธิ์สูงสุดให้ใครก็ได้ที่เปิด devtools ดู — ต้องเป็นตัวแปรฝั่ง server สำหรับ Netlify Function เท่านั้น) ใช้สำหรับฟีเจอร์ admin เพิ่มผู้ใช้งานจากในแอป และฟอร์มลีดสาธารณะ (ดูหัวข้อ "ลีดจากฟอร์มสาธารณะ" ด้านล่าง) — ตัวแปรเดียวกัน ไม่ต้องตั้งเพิ่ม
   - `GOOGLE_SERVICE_ACCOUNT_KEY`, `GOOGLE_DRIVE_ROOT_FOLDER_ID` (โฟลเดอร์หลักใบเสนอราคา), `GOOGLE_DRIVE_COMPANY_DOCS_FOLDER_ID` (โฟลเดอร์หลักเอกสารแนบบริษัท) — ใช้สำหรับมิเรอร์ไฟล์ขึ้น Google Drive อัตโนมัติ (ไม่บังคับ ถ้าไม่ตั้งจะข้ามขั้นตอนนี้ไปเงียบๆ แค่มีอัปโหลดขึ้น Google Drive ไม่สำเร็จ ไม่กระทบการบันทึกข้อมูลหลักใน Supabase) — ดูขั้นตอนตั้งค่าในหัวข้อ "ตั้งค่า Google Drive" ด้านล่าง
5. Deploy

### วิธีที่ 2: Netlify CLI
```bash
npm install -g netlify-cli
netlify login
netlify init
netlify env:set VITE_SUPABASE_URL "https://xxxx.supabase.co"
netlify env:set VITE_SUPABASE_ANON_KEY "your-anon-key"
netlify env:set SUPABASE_SERVICE_ROLE_KEY "your-service-role-key"
netlify deploy --prod
```

### ตั้งค่า Google Drive (สำหรับมิเรอร์ไฟล์ใบเสนอราคา) — ไม่บังคับ
ฟีเจอร์นี้อัปโหลดสำเนา PDF ของใบเสนอราคา (และไฟล์ที่ลูกค้าเซ็นกลับ) ขึ้น Google Drive อัตโนมัติ แยกโฟลเดอร์ปี/เดือน — เป็นแค่สำเนาสำรอง Supabase Storage ยังเป็นที่เก็บหลักเหมือนเดิม ถ้าไม่ตั้งค่าตามนี้ แอปยังใช้งานได้ปกติทุกอย่าง แค่จะไม่มีสำเนาขึ้น Drive ให้

1. เปิด [Google Cloud Console](https://console.cloud.google.com/) สร้างโปรเจกต์ใหม่ (หรือใช้โปรเจกต์เดิม) แล้วเปิดใช้งาน **Google Drive API** (เมนู APIs & Services > Library > ค้นหา "Google Drive API" > Enable)
2. ไปที่ **APIs & Services > Credentials > Create Credentials > Service Account** ตั้งชื่อตามสะดวก ไม่ต้องกำหนดสิทธิ์ระดับโปรเจกต์เพิ่ม
3. เข้าไปที่ Service Account ที่สร้าง → แท็บ **Keys > Add Key > Create new key > JSON** จะได้ไฟล์ `.json` ดาวน์โหลดลงเครื่อง — เก็บไฟล์นี้ให้ปลอดภัย (มีสิทธิ์เท่ากับรหัสผ่าน)
4. **ต้องใช้ Shared Drive เท่านั้น ห้ามใช้โฟลเดอร์ธรรมดาใน My Drive** — Service Account ไม่มีโควต้าพื้นที่จัดเก็บของตัวเอง ต่อให้แชร์สิทธิ์ Editor ให้กับโฟลเดอร์ปกติก็จะสร้างไฟล์ไม่ได้ (error "Service Accounts do not have storage quota") วิธีสร้าง: เปิด [Google Drive](https://drive.google.com) → แถบซ้าย **Shared drives** → **+ New** → ตั้งชื่อ → Create
   จากนั้นเข้าไปใน Shared Drive นั้น → **Manage members** → เปิดไฟล์ JSON หาค่า `client_email` (เช่น `xxx@xxx.iam.gserviceaccount.com`) → เพิ่มอีเมลนั้นเป็นสมาชิก สิทธิ์ **Content Manager** — จะใช้ Shared Drive เดียวกันสร้างโฟลเดอร์ย่อย 2 อัน (เช่น "Quatation" กับ "Customer Files") แยกใบเสนอราคา/เอกสารบริษัท หรือจะสร้างคนละ Shared Drive ไปเลยก็ได้ ไม่บังคับ
5. คัดลอก Folder ID ของโฟลเดอร์ปลายทางแต่ละอันจาก URL (ส่วนหลัง `/folders/` ใน `https://drive.google.com/drive/folders/<FOLDER_ID>`) — ต้องมี 2 อัน: โฟลเดอร์เก็บใบเสนอราคา และโฟลเดอร์เก็บเอกสารแนบบริษัท (จะเป็นโฟลเดอร์ย่อยใน Shared Drive เดียวกัน หรือคนละ Shared Drive ก็ได้ ขอแค่ Service Account มีสิทธิ์ Content Manager ในนั้น)
6. ตั้งค่า Netlify environment variables 3 ตัว:
   - `GOOGLE_SERVICE_ACCOUNT_KEY` = **แนะนำให้แปลงเป็น Base64 ก่อนวาง** เพราะ JSON ตรงๆ มี `\n` ฝังอยู่ใน `private_key` ถ้าวางลงช่อง Value ของ Netlify ตรงๆ มักเพี้ยนจนอ่านไม่ออก (error "ไม่ใช่ JSON ที่ถูกต้อง") วิธีแปลง เปิด Terminal บนเครื่อง Mac แล้วรัน (แทนที่ path ด้วยตำแหน่งไฟล์ `.json` ที่ดาวน์โหลดมาจริง):
     ```bash
     base64 -i /path/to/your-key.json -o /path/to/key-base64.txt
     ```
     แล้วเปิดไฟล์ `key-base64.txt` ด้วยโปรแกรมข้อความ (เช่น TextEdit) คัดลอกเนื้อหาทั้งหมดไปวางลงช่อง Value (ฟังก์ชันฝั่งเซิร์ฟเวอร์รองรับทั้ง JSON ตรงๆ และ Base64 ของ JSON อัตโนมัติ แต่ Base64 ปลอดภัยกว่าเพราะไม่มีอักขระพิเศษให้เพี้ยน) — **หมายเหตุ:** ถ้าเครื่องมีการตั้งค่าที่คัดลอกข้อความในเทอร์มินัลไปทับ clipboard อัตโนมัติ (เช่นบาง shell plugin) การใช้ `pbcopy` ต่อท้ายคำสั่งอาจโดนทับค่าไปกลายเป็นข้อความคำสั่งเอง ให้เขียนไฟล์แล้วคัดลอกจากโปรแกรมข้อความแทนจะชัวร์กว่า
   - `GOOGLE_DRIVE_ROOT_FOLDER_ID` = Folder ID ของโฟลเดอร์เก็บใบเสนอราคา จากขั้นตอนที่ 5
   - `GOOGLE_DRIVE_COMPANY_DOCS_FOLDER_ID` = Folder ID ของโฟลเดอร์เก็บเอกสารแนบบริษัท จากขั้นตอนที่ 5
7. Deploy ใหม่อีกครั้งให้ Netlify Function อ่านค่าตัวแปรใหม่

### ทดสอบฟีเจอร์ "เพิ่มผู้ใช้งาน" ในเครื่อง
`npm run dev` (Vite) ไม่รัน Netlify Function ให้ ต้องใช้ Netlify CLI แทน:
```bash
npm install -g netlify-cli
netlify dev
```
แล้วตั้งค่า `SUPABASE_SERVICE_ROLE_KEY` ไว้ในไฟล์ `.env` ของเครื่อง (Netlify CLI จะอ่านจาก `.env` ให้ฟังก์ชันด้วย) หรือรันแล้ว deploy ขึ้น Netlify จริงเพื่อทดสอบก็ได้เช่นกัน

## ย้ายข้อมูลเดิมจาก Google Sheets
ถ้ามีข้อมูลอยู่ใน Google Sheet เดิมอยู่แล้ว:
1. Export แต่ละชีทเป็น CSV (File > Download > CSV)
2. ใน Supabase → Table Editor → เลือกตารางที่ตรงกัน (เช่น `companies`) → **Insert > Import data from CSV**
3. แม็ปคอลัมน์ให้ตรงกับชื่อฟิลด์ภาษาอังกฤษในตาราง (ดูชื่อคอลัมน์ได้จาก `supabase/schema.sql`)
4. ทำทีละตารางตามลำดับ: companies → contacts/deals → activities/tasks/quotations (เพราะมี foreign key อ้างถึงกัน)

## ฟีเจอร์ที่เพิ่มเข้ามา

### เอกสารแนบ (Attachments)
- แต่ละบริษัทมีแท็บ "เอกสารแนบ" ให้อัปโหลด/ดาวน์โหลด/ลบไฟล์ได้ (จำกัดไฟล์ละไม่เกิน 20MB)
- ไฟล์เก็บใน Supabase Storage bucket ชื่อ `attachments` (private bucket, เข้าถึงผ่าน signed URL อายุ 60 วินาทีเท่านั้น)

### สิทธิ์การใช้งาน Admin / Sale / Finance
- ผู้ใช้งานใหม่ทุกคนเริ่มต้นเป็น **พนักงานขาย (sale)** อัตโนมัติ — เห็น/แก้ไข/ลบได้เฉพาะข้อมูล (บริษัท, ดีล, งาน) ที่ตัวเองสร้าง บวกกับข้อมูลเก่าที่ยังไม่มีเจ้าของ
- **ผู้ดูแลระบบ (admin)** เห็น/แก้ไข/ลบได้ทุกอย่าง และมีเมนู "ผู้ใช้งาน" ในการปรับสิทธิ์คนอื่น
- **ฝ่ายบัญชี (finance)** เพิ่มเข้ามาสำหรับโมดูลตรวจสอบยอดโอน — เห็นคำขอตรวจยอดทุกใบ (ไม่จำกัดเฉพาะของตัวเอง) และมีเมนู "ตรวจสอบยอดโอน" สำหรับอนุมัติ/ตีกลับ ส่วนข้อมูลขายอื่น (บริษัท/ดีล ฯลฯ) เห็นเท่าที่ RLS อนุญาตเหมือนพนักงานทั่วไป — บังคับจริงที่ RLS ผ่าน helper `is_finance()`
- ผู้ติดต่อ/กิจกรรม/ใบเสนอราคา/เอกสารแนบ สืบสิทธิ์ตามบริษัทแม่ (ถ้าเห็นบริษัทได้ ก็จัดการของในบริษัทนั้นได้)
- สิทธิ์ถูกบังคับที่ระดับฐานข้อมูล (Row Level Security) เสมอ — ปุ่มที่ซ่อนในหน้าเว็บเป็นแค่ UX เท่านั้น
- Admin จัดการผู้ใช้งานได้ครบจากในแอปที่หน้า "ผู้ใช้งาน":
  - **+ เพิ่มผู้ใช้งาน** — กรอกอีเมล/ชื่อ/ตั้งรหัสผ่านให้เลย ยังไม่มีอีเมลแจ้งอัตโนมัติ ต้องคัดลอกรหัสผ่านไปแจ้งพนักงานเอง
  - **แก้ไข** — แก้ชื่อ-นามสกุล/อีเมลของผู้ใช้งานคนนั้น
  - **รีเซ็ตรหัสผ่าน** — ใช้เมื่อเซลล์/บัญชีลืมรหัสผ่าน ระบบ **ดูรหัสผ่านเดิมไม่ได้เลยไม่ว่ากรณีใด** (Supabase Auth เก็บรหัสผ่านแบบ hash ทางเดียว ไม่มี API ไหนดึงค่าจริงคืนได้) ปุ่มนี้จึง**สุ่มรหัสผ่านใหม่มาตั้งแทนเลย** แล้วโชว์ให้แอดมินคัดลอกไปแจ้งพนักงาน (เห็นได้ครั้งเดียวตอนนั้น ปิดหน้าต่างแล้วดูซ้ำไม่ได้)
  - **ลบ** — ลบบัญชีถาวร (ลบตัวเองไม่ได้ กันล็อกตัวเองออกจากระบบ)
  - ทั้งหมดนี้รันผ่าน Netlify Function `manage-user` (แก้ไข/รีเซ็ตรหัสผ่าน/ลบ) และ `create-user` (เพิ่มใหม่) ที่ใช้ **Service Role Key** (สิทธิ์สูงสุด ข้าม RLS ทั้งหมด) — คีย์นี้อยู่ฝั่ง server เท่านั้น ต้องตั้งค่าเพิ่มก่อนใช้งานได้ (ดูขั้นตอนด้านล่าง) และต้องเป็น admin เท่านั้นถึงจะเรียกได้ (เช็คสิทธิ์ฝั่ง server ก่อนทำทุกครั้ง)

### Pagination
- หน้า บริษัทลูกค้า / ผู้ติดต่อ / ประวัติการติดต่อ / งาน Follow-up / ใบเสนอราคา โหลดข้อมูลทีละหน้า (20 รายการ) จากฐานข้อมูลโดยตรง แทนการโหลดทั้งหมดมาไว้ในเบราว์เซอร์
- Dashboard, ค้นหาส่วนกลาง, หน้ารายละเอียดบริษัท (แท็บย่อย) และ Kanban ดีล ยังใช้ข้อมูลชุดเต็มเหมือนเดิม เพราะต้องคำนวณสรุปข้ามทั้งระบบ — เหมาะกับข้อมูลระดับหลักพันรายการ ถ้าข้อมูลโตกว่านี้มากควรแยกไปคำนวณสรุปฝั่ง Postgres (view/RPC) แทน

### นำเข้าข้อมูลจากไฟล์ Excel (บริษัท / สินค้า / ผู้ติดต่อ)
- หน้า "บริษัทลูกค้า", "สินค้า" และ "ผู้ติดต่อ" มีปุ่ม **"นำเข้าจากไฟล์"** แบบเดียวกัน — ดาวน์โหลด Template (.xlsx) กรอกข้อมูลใน Excel แล้วอัปโหลดไฟล์กลับมาที่นี่ได้เลย (ไม่ต้อง save เป็น .csv อีกต่อไป) ระบบจะพรีวิวแถวที่ถูก/ผิดก่อนยืนยันนำเข้าเสมอ
- นำเข้าผู้ติดต่อ: template มีคอลัมน์หัวข้อ/ชื่อ-นามสกุล/โทรศัพท์ (บังคับ) และอีเมล/ตำแหน่ง/ประเภทธุรกิจ/สนใจเครื่องใช้ไฟฟ้า/เหตุผลในการซื้อ/ข้อความ/ที่มา (ไม่บังคับ) ตรงกับฟิลด์ในฟอร์มสาธารณะ `/lead` ทุกอย่าง — คอลัมน์ "สนใจเครื่องใช้ไฟฟ้า" กรอกได้หลายค่าคั่นด้วยจุลภาค (ระบบแยกเป็น array ให้อัตโนมัติ) ไม่มีการเช็คซ้ำกับข้อมูลเดิมเพราะลีดหลายรายการมาจากคนเดียวกัน/เบอร์เดียวกันได้ตามปกติ — ตัวเลือก dropdown ในไฟล์ (ตำแหน่ง/ประเภทธุรกิจ/เหตุผลในการซื้อ) ใช้ชุดเดียวกับฟอร์มสาธารณะ เก็บไว้ที่ `src/lib/leadOptions.js` ไฟล์เดียว ไม่ต้องแก้ 2 ที่ถ้าจะเปลี่ยนตัวเลือก
- ใช้ไลบรารี `exceljs` อ่าน/สร้างไฟล์ Excel (ไม่ใช้ไลบรารี `xlsx` ที่มีบน npm เพราะเวอร์ชันบน npm ค้างที่เวอร์ชันเก่าที่มีช่องโหว่ความปลอดภัยยังไม่มีแพตช์) โค้ดกลางอยู่ที่ `src/lib/importExport.js` (`readExcelRows`, `downloadExcelTemplate`, `exportRowsToExcel`) ใช้ร่วมกันทุกฟีเจอร์ import/export ในระบบ
- นำเข้าสินค้า: กันรหัสสินค้าซ้ำให้อัตโนมัติทั้งกรณีซ้ำกับที่มีอยู่แล้วในระบบ และซ้ำกันเองภายในไฟล์ที่อัปโหลด
- นำเข้าบริษัท: ไฟล์ template มีคอลัมน์ **"เลขประจำตัวผู้เสียภาษี"** ด้วย (คอลัมน์ไม่บังคับเหมือนคอลัมน์อื่นๆ ยกเว้น "ชื่อบริษัท")
- นำเข้าบริษัท: ถ้าชื่อบริษัทในไฟล์**ซ้ำกับที่มีอยู่แล้วในระบบ** ระบบจะไม่นำเข้าทันที แต่แยกแสดงเป็นรายการ "ชื่อซ้ำกับข้อมูลเดิม" ให้เลือกทีละแถวว่าจะ **"สร้างใหม่ (ซ้ำชื่อ)"** หรือ **"อัปเดตทับข้อมูลเดิม"** — ต้องเลือกให้ครบทุกแถวก่อนถึงจะกดนำเข้าได้ ส่วนชื่อที่ซ้ำกันเองภายในไฟล์เดียวกัน (ไม่ได้ซ้ำกับระบบ) ถือเป็นแถวผิดพลาดต้องแก้ในไฟล์ก่อน
- มีปุ่ม **"ส่งออกเฉพาะแถวที่มีปัญหา"** (แถวผิดพลาด + แถวชื่อซ้ำ) ดาวน์โหลดเป็นไฟล์ Excel แยกต่างหากให้ผู้อัปโหลดตรวจทานได้สะดวก โดยไม่ต้องไล่หาในไฟล์ต้นฉบับ
- Template ไฟล์บริษัทมี **dropdown ตัวเลือกฝังอยู่ในไฟล์ Excel เลย** (ไม่ต้องพิมพ์เอง กันพิมพ์ผิด) — คอลัมน์ "ประเภทลูกค้า"/"อุตสาหกรรม"/"สถานะ"/"ที่มา" ดึงค่าปัจจุบันจาก Picklists มาเป็นตัวเลือก — ค่าตัวเลือกเก็บอยู่ในชีตที่ซ่อนไว้ชื่อ "Lists" ในไฟล์ (มาจากข้อจำกัดของ Excel ที่ใส่ตัวเลือกยาวๆ ลงในสูตรตรงๆไม่ได้ถ้าเกิน 255 ตัวอักษร)
- ทุกครั้งที่เพิ่มคอลัมน์ใหม่ที่เป็น dropdown ในฟอร์ม (เช่น `customer_type`) ต้องอัปเดต template ให้มีคอลัมน์นั้นพร้อม dropdown ด้วยเสมอ ไม่ให้ template ตกรุ่นตามหลังฟอร์มจริง
- หน้า "สินค้า" มีปุ่ม **"ส่งออกเป็น Excel"** เพิ่มจากปุ่มนำเข้า — ส่งออกรหัสสินค้า/ชื่อสินค้า/วันที่เพิ่มของสินค้าทั้งหมดในระบบเป็นไฟล์ `.xlsx`

### เลือกบริษัท/สินค้าแบบพิมพ์ค้นหา (Searchable Dropdown)
- Dropdown เลือก**บริษัท**และเลือก**สินค้า** (ในฟอร์มดีล/ใบเสนอราคา/ผู้ติดต่อ/กิจกรรม/งาน) พิมพ์ชื่อหรือบางส่วนของชื่อเพื่อกรองตัวเลือกได้เลย ไม่ต้องเลื่อนหาในลิสต์ยาวๆ — component อยู่ที่ `src/components/SearchableSelect.jsx`
- ตั้งใจใช้เฉพาะ dropdown ที่ตัวเลือกอาจมีจำนวนมาก (บริษัท/สินค้า) เท่านั้น ส่วน Dropdown แบบ Picklist (อุตสาหกรรม, สถานะ ฯลฯ) ที่ตัวเลือกน้อยและแก้ไขได้เองยังใช้ dropdown ปกติเหมือนเดิม

### แนบเอกสารตอนสร้างบริษัท
- ฟอร์มเพิ่ม/แก้ไขบริษัทมีช่องแนบเอกสาร (ภพ20, หนังสือรับรองบริษัท ฯลฯ) ได้เลยตอนสร้าง/แก้ไขบริษัท — ไฟล์จะถูกเก็บในระบบเอกสารแนบเดียวกับแท็บ "เอกสารแนบ"

### รายการสินค้าในดีล + คำนวณ VAT อัตโนมัติ
- หน้า "ดีลการขาย" มีตัวกรองช่วงวันที่สร้างดีล (จาก-ถึง) ที่แถบด้านบน — ใช้ดูดีลที่สร้างขึ้นในช่วง/เดือนนั้นๆ
- เหนือ Kanban มีการ์ด **"ดีลที่ต้องติดตาม (Follow-up) แยกตามเดือน"** — สรุปดีลที่ยังไม่ปิด (ไม่ใช่ Closed Won/Lost) ที่มีวันที่ต้อง Follow up ไว้ จัดกลุ่มตามเดือน เดือนที่เลยกำหนดมาแล้วไฮไลต์สีแดงให้เห็นชัด กันดีลตกหล่น (การ์ดนี้ไม่ขึ้นกับตัวกรองช่วงวันที่สร้างด้านบน — ดูภาพรวมทั้งหมดเสมอ ไม่แสดงถ้าไม่มีดีลที่ต้องติดตาม)
- ฟอร์มเพิ่ม/แก้ไขดีลมีเพิ่ม: **ที่มาของดีล** (ไลน์/เทเลเซลล์/อีเมลล์ ฯลฯ — แก้ไขตัวเลือกได้เฉพาะ admin เหมือน dropdown อื่น), **วันที่ต้อง Follow up**, และ **รายการสินค้า** ที่กดเพิ่มได้หลายรายการ (เลือกสินค้า + จำนวน + ราคาต่อหน่วย)
- ราคาต่อหน่วยที่กรอกถือว่า **รวม VAT แล้ว** — ระบบคำนวณยอดไม่รวม VAT, ยอด VAT (7%), และยอดรวมทั้งสิ้นให้อัตโนมัติจากผลรวมทุกรายการ
- **"มูลค่า" ของดีล (ที่ใช้คำนวณยอดใน Dashboard/Kanban) ถูกคำนวณจากรายการสินค้าโดยอัตโนมัติเสมอ ไม่มีช่องให้กรอกมูลค่าเองแล้ว** — ถ้าไม่เลือกรายการสินค้าเลย มูลค่าจะเป็น 0
- เมนู "สินค้า" (อยู่ในหมวด "การขาย" เห็นได้ทุกคนที่ login) ใช้จัดการรายชื่อสินค้าที่เลือกได้ในดีล/ใบเสนอราคา — มี **รหัสสินค้า + ชื่อสินค้า + รูปสินค้า (ไม่บังคับ)** ไม่ได้เก็บราคาไว้ล่วงหน้า เพราะราคาต่อหน่วยกรอกใหม่ทุกครั้งตอนสร้างดีล/ใบเสนอราคา
- **ทุกคนที่ login แล้วเพิ่ม/แก้ไข/ลบสินค้าได้หมด** ไม่ได้จำกัดแค่ admin (ต่างจาก Picklists ที่จำกัดแค่ admin) — ตามที่ตกลงกันไว้ ถ้าอยากจำกัดสิทธิ์แจ้งได้ จะแก้ RLS policy `products write` ให้
- รูปสินค้าเก็บใน Storage bucket **"product-images"** ซึ่งเป็น **public bucket** (ต่างจาก "attachments" ที่ private) เพราะรูปสินค้าไม่ใช่ข้อมูลลับ ทำให้ใช้ URL ตรงได้เลยไม่ต้องขอ signed URL — อัปโหลด/ลบรูปได้ทุกคนที่ login เหมือนกัน, ดูได้ทุกคน (แม้ยังไม่ login ก็ได้ เพราะ bucket เป็น public)

### สรุปใบเสนอราคาแยกตามสถานะ + ตัวกรองช่วงวันที่ + แนบไฟล์ที่ลูกค้าเซ็นกลับ
- หน้า "ใบเสนอราคา" มีการ์ดสรุปจำนวน+มูลค่าแยกตามสถานะ (Draft/Sent/Approved/Rejected/Expired) อยู่เหนือตาราง — อัปเดตตามตัวกรองค้นหา/ช่วงวันที่ที่ตั้งไว้เสมอ แต่ไม่ขึ้นกับตัวกรอง "สถานะ" ตัวเดียว (ตั้งใจให้เห็นทุกสถานะพร้อมกัน)
- มีตัวกรองช่วงวันที่ใบเสนอราคา (จาก-ถึง) อิงจาก `quot_date`
- ทั้งหน้ารายการใบเสนอราคาและแท็บ "ใบเสนอราคา" ในหน้ารายละเอียดบริษัท มีปุ่ม **"แนบไฟล์เซ็นกลับ"** สำหรับอัปโหลดใบเสนอราคาที่ลูกค้าเซ็นแล้วส่งกลับมา (เก็บใน bucket "attachments" เดียวกับเอกสารแนบอื่น) — เมื่อแนบแล้วปุ่มจะเปลี่ยนเป็น "ไฟล์เซ็นกลับ" (ดาวน์โหลด) และ "ลบไฟล์เซ็นกลับ"
- ฟอร์มเพิ่ม/แก้ไขบริษัทเพิ่มช่อง **"เลขประจำตัวผู้เสียภาษี"** ใช้แสดงในใบเสนอราคาที่พิมพ์
- ใบเสนอราคาแก้ไขได้แล้ว — ปุ่ม **"แก้ไข"** เปิดฟอร์มเดิมพร้อมข้อมูลเดิมให้แก้ก่อนพิมพ์ ไม่ต้องลบทิ้งแล้วสร้างใหม่ (ทั้งหน้ารายการและแท็บในหน้าบริษัท)
- ฟอร์มใบเสนอราคารองรับ **หลายรายการสินค้าต่อใบ** เหมือนดีล — กด "+ เพิ่มรายการ" เพิ่มแถวได้ไม่จำกัด แต่ละแถวเลือกสินค้าได้ (ไม่บังคับ, เติมชื่อรายการ+โชว์รูปสินค้าอัตโนมัติ) หรือพิมพ์ชื่อรายการเองก็ได้ (เผื่อของที่ไม่มีในรายการสินค้า) พร้อมจำนวน+ราคาต่อหน่วย (ถือว่ารวม VAT แล้ว) — **"มูลค่ารวม" คำนวณจากผลรวมทุกรายการอัตโนมัติ ไม่มีช่องกรอกมูลค่าเองแล้ว** เหมือนกับดีล
- มีช่อง **"เบอร์ติดต่อเซลล์"** ต่อใบเสนอราคา ใช้แสดงในกล่องข้อมูลติดต่อตอนพิมพ์ — ใบเสนอราคาใหม่เติมเบอร์กลาง `0918086924` ให้อัตโนมัติ แก้ไข/ลบเปลี่ยนเป็นเบอร์อื่นได้เสมอ (เผื่อกรณีมีเซลล์ท่านอื่นออกใบเสนอราคาเอง หรือเบอร์เปลี่ยน)
- มีช่อง **"ชื่อผู้เสนอราคา"** (เติมชื่อผู้ใช้งานที่ล็อกอินให้อัตโนมัติตอนสร้างใบใหม่ แก้ไขได้) พิมพ์แสดงอยู่เหนือเส้น "ผู้เสนอราคา" ในเอกสารที่พิมพ์ออกมาเลย ไม่ต้องพรินซ์ออกมาเซ็นสดก่อนส่งลูกค้า
- ช่อง **"หมายเหตุ"** เติมเงื่อนไขมาตรฐานของบริษัท (การจัดส่ง/รับประกัน ฯลฯ) ให้อัตโนมัติทุกครั้งที่สร้างใบเสนอราคาใหม่ — แก้ไข/ลบข้อความได้ตามเงื่อนไขที่ตกลงกับลูกค้าจริง (เทมเพลทอยู่ใน `src/components/Modals.jsx` ค่าคงที่ `DEFAULT_QUOTATION_NOTE` แก้ข้อความเทมเพลทได้ที่นี่)
- ใบเสนอราคาเก่าก่อนมีระบบรายการสินค้า (มีแค่ subject/value เดี่ยว) จะถูกย้ายเข้าระบบรายการสินค้าให้อัตโนมัติตอนรัน schema.sql (ถ้าตอนนั้นมีการเลือกสินค้าไว้แล้ว) หรือแปลงเป็นรายการเดียวให้อัตโนมัติตอนเปิดแก้ไขครั้งแรก (ถ้าเป็นใบที่กรอกแบบเดิมไม่มีสินค้า) — ข้อมูลเดิมไม่หายแน่นอน

### รูปแบบใบเสนอราคา PDF
- ปรับรูปแบบใบเสนอราคาที่พิมพ์/บันทึกเป็น PDF ใหม่ทั้งหมด ให้ตรงกับแบบฟอร์มที่บริษัทใช้จริง — แถบหัวกระดาษสีกรมท่า (มีคำว่า QUOTATION ภาษาอังกฤษด้วย) โลโก้ + ข้อมูลบริษัท, เลขที่/วันที่, ข้อมูลลูกค้าพร้อมเลขผู้เสียภาษี, ตารางรายการสินค้าหลายแถว (พร้อมรูปสินค้าถ้ามี), กล่องสรุปยอด (ไม่รวม VAT / VAT 7% / ยอดชำระ), หมายเหตุ, และกล่องข้อมูลติดต่อ (รวมเบอร์เซลล์ถ้ากรอกไว้)

### มิเรอร์ไฟล์ใบเสนอราคา + เอกสารแนบบริษัทขึ้น Google Drive อัตโนมัติ

- โครงสร้างโฟลเดอร์เอกสารแนบบริษัทใน Drive: `<โฟลเดอร์หลักเอกสารบริษัท> / <ชื่อบริษัท> /` — สร้างโฟลเดอร์ตามชื่อบริษัทอัตโนมัติถ้ายังไม่มี
- ชื่อไฟล์ใบเสนอราคาตั้งตามเลขที่เสมอ เช่น `QT26070006.pdf` (ตัวใบเสนอราคาเอง) และ `QT26070006_Sign.pdf` (ไฟล์เซ็นกลับ — นามสกุลไฟล์ตามไฟล์ที่ลูกค้าอัปโหลดจริง เช่น `.jpg` ถ้าเป็นรูปสแกน) ส่วนเอกสารแนบบริษัทใช้ชื่อไฟล์เดิมตามที่อัปโหลด — แก้ไข/อัปโหลดซ้ำจะเขียนทับไฟล์เดิมที่ชื่อเดียวกัน ไม่สร้างไฟล์ซ้ำซ้อน (เก็บ Drive file id ไว้ในคอลัมน์ `quotations.drive_file_id`/`drive_signed_file_id`/`attachments.drive_file_id`)
- ต้องตั้งค่า Service Account ของ Google ก่อนถึงจะใช้งานได้ (ดูหัวข้อ "ตั้งค่า Google Drive" ด้านบน) — ถ้ายังไม่ตั้งค่า ฟีเจอร์อื่นๆ ในแอปใช้งานได้ปกติ แค่จะไม่มีสำเนาขึ้น Drive
- สร้าง PDF ด้วยการเรนเดอร์เทมเพลตในกล่องที่ซ่อนไว้ในหน้าเว็บแล้วถ่ายภาพด้วย `html2canvas` ต่อด้วย `jsPDF` (แบ่งหน้า A4 อัตโนมัติถ้ารายการสินค้ายาว) — โค้ดอยู่ที่ `src/lib/printQuotation.js` (`renderQuotationPdfBlob`) ส่วนการอัปโหลดขึ้น Drive จริงอยู่ที่ Netlify Function `netlify/functions/upload-drive-file.js` เดียวกันทั้งสองกรณี (ต้อง login ก่อนถึงจะเรียกได้ ไม่เปิดสาธารณะ) รับ `folderPath` (array ของชื่อโฟลเดอร์ไล่ตามลำดับ) และ `purpose` (`'company-doc'` ใช้โฟลเดอร์หลักเอกสารบริษัท ค่าอื่น/ไม่ระบุใช้โฟลเดอร์หลักใบเสนอราคา) เพื่อเลือกโฟลเดอร์หลักที่ถูกต้อง

### เชื่อมข้อมูลระหว่างดีลกับใบเสนอราคา (ไม่ต้องกรอกซ้ำสองที่)
- ปุ่ม **"ออกใบเสนอราคา"** ที่การ์ดดีล (หน้า Kanban ดีลการขาย และแท็บ "ดีล" ในหน้าบริษัท) — เปิดฟอร์มใบเสนอราคาใหม่พร้อมก็อปบริษัท/หัวข้อ/รายการสินค้าทั้งหมดจากดีลมาให้ทันที ไม่ต้องพิมพ์ซ้ำ กดได้ไม่จำกัดจำนวนครั้ง (ดีลเดียวออกใบเสนอราคาได้หลายใบ)
- ปุ่ม **"สร้างดีล"** ที่แถวใบเสนอราคา (หน้า "ใบเสนอราคา" และแท็บ "ใบเสนอราคา" ในหน้าบริษัท) — เปิดฟอร์มดีลใหม่พร้อมก็อปบริษัท/รายการสินค้าจากใบเสนอราคามาให้ ปุ่มนี้จะไม่แสดงถ้าใบเสนอราคานั้นผูกกับดีลอยู่แล้ว (กันสร้างดีลซ้ำจากใบเดียวกันโดยไม่ตั้งใจ)
- ทั้งสองฝั่งผูกกันด้วยคอลัมน์ `quotations.deal_id` (มีอยู่ในสคีมาเดิมแล้วแต่ไม่เคยถูกใช้งาน) — เมื่อผูกแล้วจะเห็น badge เล็กๆ ทั้งสองทาง เช่น ที่การ์ดดีลโชว์ "ออกใบเสนอราคาแล้ว N ใบ" และที่แถวใบเสนอราคาโชว์ "จากดีล: ชื่อดีล"
- **คัดลอกครั้งเดียว ไม่ sync ต่อเนื่อง** — กดปุ่มแล้วรายการสินค้าจะถูกก็อปเป็นชุดใหม่ไปยังอีกฝั่ง แก้ไขต่อได้อิสระทั้งสองฝั่งโดยไม่กระทบกัน (เช่น ดีลมีสินค้า A แต่ตอนออกใบเสนอราคาลูกค้าขอเพิ่มสินค้า B ก็เพิ่มแถวในฟอร์มใบเสนอราคาต่อได้เลย ดีลเดิมไม่ถูกแก้ตาม)
- โครงสร้างรายการสินค้าของดีล (`deal_items`) และใบเสนอราคา (`quotation_items`) เหมือนกันเป๊ะ (`product_id` + `description` + `quantity` + `unit_price`) ก็อปไปมาได้ตรงๆ ไม่มีข้อจำกัด รวมถึงแถวที่พิมพ์ชื่อรายการเอง ไม่ได้เลือกจากรายการสินค้าในระบบ ก็ก็อปได้ปกติ

### คัดลอกใบเสนอราคา (ลูกค้าเดิมกลับมาซื้อซ้ำ)
- ปุ่ม **"คัดลอก"** ที่แถวใบเสนอราคา (หน้า "ใบเสนอราคา" และแท็บ "ใบเสนอราคา" ในหน้าบริษัท) — เปิดฟอร์มใบเสนอราคาใหม่พร้อมก็อปบริษัท/หัวข้อ/รายการสินค้า/หมายเหตุ/เบอร์ติดต่อเซลล์/ชื่อผู้เสนอราคา/เงื่อนไขเครดิตจากใบเดิมมาให้ทันที เหมาะกับลูกค้าเก่าที่กลับมาสั่งสินค้าเดิมซ้ำ ไม่ต้องพิมพ์ใหม่ทั้งหมด

### ลูกค้าบุคคลธรรมดา / นิติบุคคล
- ฟอร์มเพิ่ม/แก้ไข **"ลูกค้า"** (ยังใช้ตาราง `companies` เดียวกันเหมือนเดิม ไม่ได้แยกตารางใหม่ — เลือกได้จาก dropdown เดียวกันตอนสร้างดีล/ใบเสนอราคาเหมือนกันหมด) มีช่อง **"ประเภทลูกค้า"** ให้เลือก **นิติบุคคล/บริษัท** (ฟอร์มเต็มแบบเดิม) หรือ **บุคคลธรรมดา** (ฟอร์มย่อ — ซ่อนช่ออุตสาหกรรม/เว็บไซต์/เลขประจำตัวผู้เสียภาษีที่ไม่เกี่ยวกับบุคคล)
- เหมาะกับลูกค้าที่ซื้อสินค้าไปใช้ส่วนตัว ไม่มีข้อมูลบริษัท แต่ยังต้องการใบเสนอราคา — กรอกแค่ชื่อ-เบอร์-ที่อยู่สั้นๆ แล้วเลือกได้จาก dropdown ตอนสร้างดีล/ใบเสนอราคาเหมือนลูกค้าบริษัททุกอย่าง ไม่ต้องพิมพ์ข้อมูลซ้ำมือทุกครั้ง
- หน้า "บริษัทลูกค้า" มีตัวกรอง "ประเภทลูกค้า" เพิ่ม และมี badge เล็กๆ "บุคคล" กำกับไว้ที่ชื่อในรายการ ให้แยกจากลูกค้านิติบุคคลได้ง่าย
- คอลัมน์ "ประเภทลูกค้า" ในไฟล์ template นำเข้าบริษัทลูกค้าก็มี dropdown ตัวเลือกให้เหมือนคอลัมน์อื่นๆ (ดูหัวข้อ "นำเข้าข้อมูลจากไฟล์ Excel" ด้านบน)
- ฟอร์มดีล/ใบเสนอราคาเลือกลูกค้าจาก dropdown เดิม (ไม่มีปุ่มเพิ่มลูกค้าใหม่ในฟอร์มนี้ — ต้องไปเพิ่มลูกค้าที่หน้า "บริษัทลูกค้า" ก่อนถ้ายังไม่มีในระบบ) แต่เพิ่มช่อง **"ประเภทลูกค้า"** ไว้ด้านบน dropdown เลือกบริษัท ใช้เป็นตัวกรองช่วยแคบรายการก่อนค้นหา (เลือก "บุคคลธรรมดา" หรือ "นิติบุคคล/บริษัท" แล้ว dropdown บริษัทด้านล่างจะโชว์แค่ลูกค้ากลุ่มนั้น ไม่เลือกก็เห็นทุกกลุ่มเหมือนเดิม) — ไม่ต้องกรอกข้อมูลลูกค้าใหม่ เพราะดึงข้อมูลลูกค้าที่มีอยู่แล้วมาให้เลือก/ค้นหาได้ทันที ไม่ว่าจะเป็นบุคคลธรรมดาหรือนิติบุคคล

### ผู้ติดต่อ (ลีดจากฟอร์มสาธารณะ Facebook/เว็บไซต์) + แปลงเป็นลูกค้า
- เมนู "ผู้ติดต่อ" เดิม (รายชื่อคนติดต่อที่ผูกกับบริษัทลูกค้าโดยตรง) ถูกเอาออกจากหน้าหลักแล้ว — การจัดการผู้ติดต่อของแต่ละบริษัทยังทำได้ปกติผ่านแท็บ "ผู้ติดต่อ" ในหน้ารายละเอียดบริษัท ไม่ได้หายไปไหน แค่ไม่มีหน้ารายการกลางแยกอีกต่อไป
- เมนู **"ผู้ติดต่อ"** ในปัจจุบัน คือหน้ารวมลีดที่เข้ามาจากฟอร์มสาธารณะ (เดิมชื่อ "ลีดที่เข้ามา") — มีฟอร์มสาธารณะที่ **`/lead`** (เช่น `https://yourapp.netlify.app/lead?src=facebook`) ไม่ต้อง login เอาลิงก์ไปแปะใน Facebook Ads หรือเว็บไซต์ได้เลย ที่มา (`source`) ถูกแท็กอัตโนมัติจาก query param `?src=` ในลิงก์ ไม่ต้องให้ลูกค้ากรอกเอง
- ฟอร์มจัดกลุ่มเป็น 3 ส่วนเรียงตามความสำคัญ: **ข้อมูลติดต่อ** (ชื่อ-นามสกุล/เบอร์โทรศัพท์ บังคับ, อีเมลไม่บังคับ) → **เรื่องที่ต้องการติดต่อ** (หัวข้อ บังคับ) → **ข้อมูลเพิ่มเติม** (ไม่บังคับทั้งหมด ช่วยให้เซลล์ติดต่อกลับได้ตรงจุดขึ้น): ตำแหน่ง, ประเภทธุรกิจ (มีตัวเลือก "อื่นๆ โปรดระบุ" ที่เด้งช่องให้พิมพ์เอง), ประเภทเครื่องใช้ไฟฟ้าที่สนใจ (เลือกได้หลายข้อ แบบ checkbox), เหตุผลในการซื้อ, และข้อความเพิ่มเติม — ตัวเลือกของ dropdown/checkbox เหล่านี้ฮาร์ดโค้ดไว้ในโค้ด (`src/components/PublicLeadPage.jsx`) ไม่ได้ผูกกับระบบ Picklists เพราะเป็นฟอร์มสาธารณะที่ยังไม่ได้ login เข้าไม่ถึงตาราง `picklists` (มี RLS จำกัดไว้เฉพาะผู้ใช้ที่ login) — ถ้าจะแก้ตัวเลือกต้องแก้โค้ดตรงนี้โดยตรง
- ฟิลด์ **"สนใจสินค้า / รุ่นไหน"** แบบพิมพ์อิสระถูกถอดออกจากฟอร์มแล้ว (แทนที่ด้วย "ประเภทเครื่องใช้ไฟฟ้าที่สนใจ" ด้านบน) แต่คอลัมน์ `leads.interested_product` เดิมยังเก็บไว้ในฐานข้อมูลเผื่อดูข้อมูลลีดเก่าที่เคยกรอกไว้ก่อนหน้า
- ข้อมูลที่กรอกจะถูกส่งผ่าน **Netlify Function `submit-lead`** (ใช้ Service Role Key เขียนเข้า Supabase แทน — ดูตัวแปร `SUPABASE_SERVICE_ROLE_KEY` ด้านบน) **ไม่ได้เปิด RLS ให้คนนอก (anon) insert เข้าตาราง `leads` ตรงๆ** เพื่อกันสแปม/ข้อมูลมั่วเขียนเข้าระบบได้ง่ายเกินไป
- หน้า "ผู้ติดต่อ" ให้เซลล์เห็นรายชื่อลีดที่กรอกเข้ามา (หัวข้อ/ชื่อ/เบอร์/ตำแหน่ง-ธุรกิจ/สนใจ/เหตุผล/ที่มา) พร้อมตัวกรองสถานะ/ค้นหา/ช่วงวันที่ (`created_at` — ใช้ดูว่าเดือนไหนมีคนกรอกเข้ามาเท่าไหร่), ปุ่ม **"ส่งออกเป็น Excel"** (ส่งออกตามตัวกรองที่ตั้งไว้ ไม่จำกัดแค่หน้าที่กำลังดูอยู่ เผื่อเอาไปใช้ต่อกับระบบอื่น รวมคอลัมน์ใหม่ทั้งหมดด้วย) และปุ่ม **"สร้างเป็นลูกค้า"** — กดแล้วก็อปหัวข้อ/ชื่อ/เบอร์/อีเมล/ข้อความไปตั้งต้นฟอร์มลูกค้าใหม่ให้ (ตั้งประเภทเป็น **บุคคลธรรมดา** โดย default เพราะลีดส่วนใหญ่คือคนซื้อไปใช้เอง แก้เป็นนิติบุคคลได้ถ้าจริงๆเป็นบริษัท) แก้ไข/เติมข้อมูลที่ลีดกรอกมาไม่ครบได้ก่อนบันทึกจริง
- เหนือตารางมีการ์ดสรุป 2 ชุด ตามช่วงวันที่/คำค้นหาที่ตั้งไว้ (ไม่ขึ้นกับตัวกรองสถานะ เพื่อให้เห็นภาพรวมทุกสถานะ/ช่องทางพร้อมกันเสมอ): **สรุปตามสถานะ** (เรียงตามลำดับใน picklist `lead_statuses`) และ **สรุปตามช่องทางที่มา** (`source` เช่น facebook/website ที่แท็กมาจากลิงก์ — ช่องทางที่ไม่มีการแท็ก `?src=` มาจะรวมอยู่ใน "ไม่ระบุที่มา")
- ปุ่ม **"+ เพิ่มผู้ติดต่อ"** ให้เซลล์กรอกผู้ติดต่อเข้าระบบเองได้โดยไม่ต้องผ่านฟอร์มสาธารณะ — ใช้กรณีลูกค้าทักมาเอง (โทร/แชท) หรือเซลล์ออกไปพบลูกค้าข้างนอกแล้วได้นามบัตร/ข้อมูลมา ฟอร์มมีฟิลด์เดียวกับฟอร์มสาธารณะทุกอย่าง (component `LeadModal` ใน `src/components/Modals.jsx`) ต่างกันที่สถานะเริ่มต้นเป็น **"ติดต่อแล้ว"** (ไม่ใช่ "ใหม่" เหมือนฟอร์มสาธารณะ) เพราะเซลล์คุยกับลูกค้าไปแล้วก่อนจะมากรอก และมีช่อง "ที่มา"/"สถานะ" ให้เลือกเองได้ทันที (ฟอร์มสาธารณะไม่มีเพราะเซลล์ยังไม่รู้จนกว่าจะติดต่อ)
- เมื่อแปลงเป็นลูกค้าแล้ว ปุ่ม "สร้างเป็นลูกค้า" จะเปลี่ยนเป็นปุ่มลิงก์ไปหน้าลูกค้าที่แปลงมาแล้วแทน (กันกดสร้างซ้ำจากลีดเดียวกัน) — เหมือน pattern การผูก deal↔quotation ด้านบน คือก็อปครั้งเดียวแล้วแก้ไขอิสระต่อได้ ไม่ sync ต่อเนื่อง
- ลบลีดได้เฉพาะ admin (สถานะ/แปลงเป็นลูกค้าทำได้ทุกคนที่ login เพราะลีดไม่มีเจ้าของเฉพาะคน เซลล์คนไหนก็ตามต่อได้)

### Dropdown แก้ไขได้เอง (Picklists)
- Dropdown ทุกตัวที่ใช้กรอกข้อมูล (อุตสาหกรรม, สถานะบริษัท, Stage ดีล, ประเภทกิจกรรม, ลำดับความสำคัญ/สถานะงาน, สถานะใบเสนอราคา, ที่มาลูกค้า, ที่มาของดีล) มีปุ่ม **"แก้ไข"** ข้างๆ ให้เพิ่ม/ลบตัวเลือกได้ทันทีโดยไม่ต้องไปหน้าตั้งค่าแยก — คล้าย dropdown list ที่แก้ไขได้ใน Google Sheets
- **เฉพาะผู้ดูแลระบบ (admin)** เท่านั้นที่เห็นปุ่ม "แก้ไข" และแก้ไข/เพิ่ม/ลบตัวเลือกได้ — พนักงานขายเลือกจากตัวเลือกที่มีอยู่ได้อย่างเดียว บังคับจริงที่ RLS policy `picklists write` (ทุกคน login แล้วยังอ่าน/เลือกตัวเลือกได้ปกติผ่าน `picklists select`)
- ⚠️ **ข้อควรระวัง:** ค่าบางตัวถูกผูกกับ logic ในโค้ดตรงๆ (`Closed Won`/`Closed Lost` ใช้คำนวณยอดดีลที่ปิดสำเร็จ, `เสร็จสิ้น` ใช้เช็คงานที่เสร็จแล้ว, `Active` ใช้นับบริษัท Active ใน Dashboard) ถ้าลบหรือเปลี่ยนชื่อค่าพวกนี้ ตัวเลขสรุปที่เกี่ยวข้องจะไม่ถูกต้องจนกว่าจะรีแฟกเตอร์โค้ดส่วนนั้นเพิ่ม (ตามที่ตกลงกันว่ายอมรับความเสี่ยงนี้ไว้ก่อน)
- รายการเริ่มต้นทั้งหมดถูก seed มาจากค่าเดิมที่เคยฮาร์ดโค้ดไว้ในโค้ด (ไม่มีอะไรหายไป) เก็บอยู่ในตาราง `picklists` ใหม่ — หน้า "ที่มาลูกค้า" เดิมถูกลบออกแล้ว เพราะปุ่ม "แก้ไข" ทำหน้าที่แทนได้ในตัว

### ลูกค้าเครดิต + ตามเก็บเงิน
- ในฟอร์มลูกค้ามีช่อง **"เงื่อนไขเครดิต"** (dropdown จาก picklist `credit_terms`: เครดิต 15/30/45/60 วัน, ว่าง = ลูกค้าเงินสด) — แค่จำแนกประเภทลูกค้าไว้เป็นค่าเริ่มต้น แสดงเป็น badge สีส้มทุกที่ที่เกี่ยวข้อง: รายการบริษัท, หน้ารายละเอียดบริษัท, การ์ดดีลใน Kanban
- ตอนสร้าง/แก้ไขใบเสนอราคา ด้านบนสุดของฟอร์มมีปุ่มสลับ **"ใบเสนอราคาแบบธรรมดา"** / **"ลูกค้าเครดิต"** คู่กับช่อง **"วันที่"** ที่ย้ายมาไว้ชิดขวาบนสุด (เหมือนตำแหน่งวันที่บนใบเสนอราคาที่พิมพ์จริง) — ถ้าเลือกบริษัทที่ตั้งเงื่อนไขเครดิตไว้ในโปรไฟล์ ระบบจะสลับเป็นโหมดเครดิตให้อัตโนมัติ (แก้กลับเป็นธรรมดาได้ถ้าใบนี้ไม่ต้องการ)
- เลือกโหมด **"ลูกค้าเครดิต"** แล้วจะมีช่องเพิ่ม: **"จำนวนวันเครดิต"** (picklist เดียวกับของบริษัท แต่เลือกแยกได้ต่อใบ เผื่อใบนี้เงื่อนไขต่างจากปกติ) และ **"วันครบกำหนดชำระ"** ที่เซลล์กรอกเองตามรอบจ่ายจริง (ไม่คำนวณอัตโนมัติจากจำนวนวันเครดิต เพราะรอบจ่ายจริงอาจสั้น/ยาวกว่าเทอมที่ตกลงกันไว้) พร้อม **"สถานะการชำระ"** (ยังไม่ชำระ/ชำระแล้ว) — ค่าที่เลือกจะพิมพ์โชว์อยู่ในใบเสนอราคา PDF ด้วย (มุมบนขวา คู่กับวันที่/เลขที่) ให้ลูกค้าเห็นเงื่อนไขชัดเจน
- ค่าเครดิตของใบเสนอราคาแต่ละใบเก็บแยกจากเงื่อนไขเครดิตของบริษัท เพื่อให้ใบเก่ายังคงข้อมูล ณ วันที่ออกไว้ ถึงแม้ภายหลังจะไปแก้เงื่อนไขเครดิตของบริษัทในโปรไฟล์
- หน้า **"ใบเสนอราคา"** มีการ์ดสรุป **"ต้องตามเก็บเงิน (ลูกค้าเครดิต)"** อยู่บนสุด แสดงทุกใบที่ยังไม่ชำระและมีวันครบกำหนดแล้ว เรียงใกล้ครบกำหนดสุดก่อน (ไม่ขึ้นกับตัวกรอง/ค้นหาบนหน้าจอ เพื่อให้เห็นภาพรวมเสมอ) วันที่ครบกำหนดจะขึ้นสีแดงถ้าเลยกำหนดแล้ว หรือสีเหลืองถ้าครบกำหนดวันนี้ — ตารางหลักและแท็บใบเสนอราคาในหน้ารายละเอียดบริษัทก็มีคอลัมน์ "ครบกำหนดชำระ"/"การชำระ" พร้อมไฮไลต์แถวสีแดงถ้าเลยกำหนดเหมือนกัน
- หน้า **"แดชบอร์ด"** มีการ์ด KPI **"ต้องตามเก็บเงิน"** เพิ่มในแถวบนสุด และการ์ดรายการ **"ต้องตามเก็บเงิน (ลูกค้าเครดิต)"** ท้ายหน้า สรุปภาพรวมใบเสนอราคาที่ต้องตามเก็บเงินทั้งระบบให้เห็นทันทีโดยไม่ต้องเข้าไปหน้าใบเสนอราคา
- ตัวกรองช่วงวันที่บนหน้ารายการ (ผู้ติดต่อ/ดีลการขาย/ใบเสนอราคา) ถูกจัดให้ชิดขวาของแถบตัวกรองเสมอ เพื่อให้ตำแหน่งตัวกรองวันที่สอดคล้องกันทุกหน้า

### ตรวจสอบยอดโอน + อนุมัติจากฝ่ายบัญชี (Payment Verification)
- โมดูลตรวจยอดโอนก่อนเปิดออเดอร์ — เปลี่ยน flow เดิม (โอน → ทำไฟล์ → ส่งบัญชี → เซ็น → เปิดออเดอร์) มาอยู่ในระบบ: **Sale สร้างคำขอตรวจยอด → แนบสลิป + รายการสินค้า → บัญชีตรวจในระบบ → อนุมัติ/ตีกลับ → Sale เปิดออเดอร์** ทุกขั้นตอนบันทึกลง `audit_logs`
- เพิ่ม 2 เมนูในกลุ่ม **"การเงิน"**: **คำขอตรวจยอด** (Sale สร้าง/ติดตาม), **ตรวจสอบยอดโอน** (เฉพาะ finance/admin)
- สถานะคำขอ (`payment_requests.status`) เป็น workflow คงที่ในโค้ด (ไม่ใช่ picklist): `Draft → Pending Finance Review → (Need More Info / Payment Mismatch / Rejected / Approved to Create Order) → Order Created` — label ไทย + สี badge อยู่ที่ `format.js` (`paymentStatusLabel`/`paymentBadgeClass`)
- **แนวคิดยอดเงิน:** ฟอร์มไม่ให้กรอกยอดที่ต้องชำระ/ยอดโอนจริง/ธนาคาร/วันเวลาโอน — บัญชีเทียบเอาเองจาก **รายการสินค้า (รวม VAT 7% แยกให้ดู)** เทียบกับ **สลิปจริง** ที่แนบมา; `total_amount` คำนวณจากรายการสินค้าโดยอัตโนมัติ (สูตร VAT เดียวกับดีล/ใบเสนอราคา ราคาต่อหน่วยรวม VAT แล้ว)
- **ขั้นตอนสร้างคำขอ (เรียงตามลำดับที่กรอกจริง):** (1) เลือกปุ่ม **ประเภทลูกค้า** (ลูกค้าเครดิต/ลูกค้าธรรมดา) ก่อน — ใช้กรองรายการใบเสนอราคาให้หาง่ายขึ้น (2) เลือก **เลขที่ใบเสนอราคา** จากรายการที่กรองแล้ว — เลือกเสร็จระบบจะคัดลอก **ประเภทลูกค้า, บริษัท, รายการสินค้า, จำนวน, ราคา** จากใบเสนอราคานั้นมาให้อัตโนมัติทั้งหมด (3) เซลล์เลือก **ประเภทการชำระ**, กรอก **Bill No.** (เลขที่บิลของออเดอร์ที่ต้องเปิดในระบบ), **เลขที่ PO**, และแนบ **สลิปการโอน** — ฟอร์มไม่มีช่องเลือกดีลอีกต่อไป (ดีล query ยังอยู่ใน DB เผื่อข้อมูลเก่า แต่ฟอร์มไม่ใช้แล้ว)
- **กฎสำคัญ:** ต้องเลือกประเภทลูกค้า + ใบเสนอราคา + มีสลิป + รายการสินค้า ≥ 1 ก่อน Submit, Submit แล้ว Sale แก้ไม่ได้จนกว่าบัญชีจะตีกลับ (Need More Info / Payment Mismatch), ตอนอนุมัติบัญชีต้อง **ระบุชื่อผู้อนุมัติ (ลายเซ็น)** และใส่ **Ref No.** (`finance_ref_no`) ได้ (ไม่บังคับ), อนุมัติแล้วระบบออก `PAY-APP-xxxxxx` อัตโนมัติ, เปิดออเดอร์ได้เฉพาะใบที่อนุมัติแล้ว — กฎ workflow เหล่านี้บังคับที่ฝั่งแอป (functions ใน `api.js`)
- **สิทธิ์เห็น/แก้ไข:** ทุกคนที่ login **เห็น**คำขอตรวจยอดได้ทุกใบเสมอ (ไม่จำกัดเฉพาะของตัวเอง — กันเซลล์มองไม่เห็นคำขอที่เพื่อนร่วมทีม/แอดมินสร้างไว้) แต่**แก้ไข/ลบ/ส่งให้บัญชี/เปิดออเดอร์**ทำได้เฉพาะเจ้าของคำขอ, finance, หรือ admin เท่านั้น (บังคับจริงที่ RLS, หน้าเว็บซ่อนปุ่มเหล่านี้ไปเลยถ้าไม่ใช่เจ้าของเพื่อไม่ให้กดแล้วเจอ error)
- **หน้าตรวจสอบยอดโอน** ค่าเริ่มต้นแสดง **ทุกสถานะ** (ไม่ได้กรองเฉพาะ "รอบัญชีตรวจ" เหมือนก่อน) เลือกกรองเองได้จาก dropdown
- **หลังอนุมัติแล้ว** หน้า "คำขอตรวจยอด" ของ Sale จะมีปุ่ม **"ดาวน์โหลด PDF"** (แสดงเฉพาะสถานะอนุมัติแล้ว/เปิดออเดอร์แล้ว) — เปิดหน้าต่างใหม่แสดง **ใบอนุมัติตรวจสอบยอดโอน** (เลขคำขอ/เลขอนุมัติ/ลูกค้า/รายการสินค้า/ผู้อนุมัติ **+ รูปสลิปที่แนบไว้ฝังในเอกสารด้วย**) ให้กด "พิมพ์ / บันทึกเป็น PDF" แล้วนำไปแนบตอนเปิดออเดอร์ในระบบอื่น — โค้ดอยู่ที่ `src/lib/printPaymentApproval.js` (รูปแบบเดียวกับ `printQuotation.js`)
  - สลิปที่เป็นรูปภาพ (jpg/png ฯลฯ) ฝังแสดงตรงๆ ในเอกสาร, ถ้าเป็น PDF จะฝังผ่าน iframe (พร้อมลิงก์สำรองเผื่อเบราว์เซอร์บางตัวไม่ยอมพิมพ์เนื้อหาใน iframe)
- ทั้งหน้า **คำขอตรวจยอด** และ **ตรวจสอบยอดโอน** มีปุ่ม **"ส่งออกข้อมูล"** เป็นไฟล์ Excel (ตามตัวกรองปัจจุบัน) ให้บัญชีเอาไปแมทช์กับระบบบัญชีภายหลัง
- ตาราง/ฟังก์ชัน: `payment_requests` (+ คอลัมน์ `request_date`/`credit_type`/`total_amount`/`finance_ref_no`/`bill_no`), `payment_items`, `audit_logs`, `notifications`, `gen_pr_no()`/`gen_approval_ref_no()`, picklist `payment_types`, helper `is_finance()` — **ต้องรัน SQL migration ก่อนใช้งาน** (ดู `supabase/schema.sql`)
- สลิปการโอนเก็บใน bucket `attachments` เดิม (โฟลเดอร์ `payment-slips/`) ใช้ signed URL เปิดดูเหมือนเอกสารแนบอื่น

### เอกสารบัญชี (Accounting Document Request & Upload Center)
- โมดูลให้เซลล์ขอเอกสาร (ใบแจ้งหนี้/ใบกำกับภาษี/ใบเสร็จ) ผ่านระบบแทนแจ้งบัญชีทีละออเดอร์ทางแชท/ปากเปล่า และให้บัญชีอัปโหลด PDF ให้เซลล์ดาวน์โหลดไปส่งลูกค้าเอง
- **หมายเหตุการปรับให้เข้ากับระบบเดิม:** requirement ต้นฉบับอ้างถึงตาราง "ออเดอร์" ที่ระบบนี้ไม่มี — จึงผูก `order_id` เข้ากับ **`quotations`** แทน (ใบเสนอราคาที่ปิดขายแล้วถือเป็น "ออเดอร์" ในทางธุรกิจของระบบนี้) และเนื่องจากไม่มีหน้ารายละเอียดออเดอร์แยกต่างหาก ปุ่ม "เอกสารบัญชี" จึงอยู่ในหน้า **"คำขอตรวจยอด"** ต่อแถวที่มีสถานะ **เปิดออเดอร์แล้ว** (`Order Created`) แทน — เปิดแล้วเป็นป็อปอัป ไม่ใช่หน้าแยก
- **ฝั่งเซลล์:** กดปุ่ม "เอกสารบัญชี" → เลือก "ต้องการเอกสารไหม" ถ้าต้องการเลือกประเภทเอกสาร/วิธีส่ง/ความเร่งด่วน + กรอกข้อมูลภาษี/อีเมล/ที่อยู่ตัวจริงตามเงื่อนไข → ถ้าข้อมูลครบระบบส่งเข้าคิวบัญชีอัตโนมัติ (ไม่ครบจะค้างสถานะ "รอข้อมูลจากเซลล์" ให้กลับมาแก้) → รอบัญชีอนุมัติ/อัปโหลดไฟล์ → เซลล์ดาวน์โหลด + คัดลอกข้อความส่งลูกค้า + Mark ว่าส่งให้ลูกค้าแล้ว (เลือกช่องทาง email/line/whatsapp/manual/other)
- **ฝั่งบัญชี:** เมนู **"เอกสารบัญชี"** (เฉพาะ finance/admin) — การ์ดสรุป 8 ตัว (รอข้อมูลจากเซลล์/รอบัญชีตรวจสอบ/รอออกเอกสาร/รออัปโหลดเอกสาร/เอกสารพร้อมดาวน์โหลด/งานด่วน/งานเกินกำหนด/เสร็จวันนี้ — คลิกการ์ดกรองตารางได้), ตัวกรองสถานะ/ความเร่งด่วน/ค้นหา/ช่วงวันที่ส่งคำขอ, ตารางครบทุกคอลัมน์ตาม requirement (รวมคอลัมน์เลขที่เอกสาร 3 ประเภทไว้ในช่องเดียวแบบ stacked เพื่อไม่ให้ตารางกว้างเกินไป)
- **workflow สถานะคงที่ในโค้ด** (`accounting_document_requests.document_status`, ไม่ใช่ picklist): `รอข้อมูลจากเซลล์ → รอบัญชีตรวจสอบ → รอออกเอกสาร → รออัปโหลดเอกสาร → เอกสารพร้อมดาวน์โหลด → (ส่งให้ลูกค้าแล้ว →) รอส่งตัวจริง → ส่งตัวจริงแล้ว → เสร็จสิ้น` (ยกเลิกได้ทุกจุดก่อนเสร็จสิ้น) — บัญชีตรวจสอบไม่ผ่านจะตีกลับเป็น "รอข้อมูลจากเซลล์" พร้อมเหตุผล
- **เวอร์ชันไฟล์:** อัปโหลดไฟล์ประเภทเดิมซ้ำไม่ลบของเก่า — ไฟล์เก่าเปลี่ยนเป็น `is_current=false`, ไฟล์ใหม่ `version_no` เพิ่มขึ้นและเป็น current แทน เซลล์เห็น/ดาวน์โหลดได้เฉพาะไฟล์ current, บัญชี/admin ดูประวัติทุกเวอร์ชันได้ในป๊อปอัป Upload
- **สิทธิ์:** Sale เห็นคำขอ/ไฟล์ได้ทุกรายการเหมือนโมดูลอื่น (RLS select เปิดกว้าง) แต่ **สร้าง/แก้ไขคำขอได้เฉพาะของตัวเอง** (หรือ finance/admin) และ **ห้าม Upload/Replace/Delete ไฟล์เอกสารบัญชีเด็ดขาด** (RLS เขียนตาราง `accounting_document_files` จำกัดเฉพาะ finance/admin เท่านั้น ไม่มีข้อยกเว้นแม้เจ้าของคำขอ) — ฟังก์ชัน SECURITY DEFINER 2 ตัว (`mark_doc_file_downloaded`, `mark_doc_file_sent_to_customer`) เปิดให้ Sale ปั๊ม timestamp "ดาวน์โหลดแล้ว"/"ส่งลูกค้าแล้ว" ได้แบบจำกัดเฉพาะคอลัมน์นั้นๆ โดยไม่ต้องมีสิทธิ์ UPDATE เต็มแถว
- ไฟล์เก็บใน bucket ใหม่ **`accounting-documents`** (private เพราะมีข้อมูลภาษี/ลูกค้า — ต่างจาก `product-images` ที่ public) path รูปแบบ `orders/{quotation_id}/{file_type}_{document_no}_v{version_no}.{ext}` เปิดด้วย signed URL อายุ 60 วินาที
- ตาราง/ฟังก์ชันใหม่: `accounting_document_requests`, `accounting_document_files`, `mark_doc_file_downloaded()`, `mark_doc_file_sent_to_customer()` — **ต้องรัน SQL migration ก่อนใช้งาน** (ดู `supabase/schema.sql`)
- **ข้อจำกัด/สิ่งที่ยังไม่ทำ:** ไม่มี role "Manager" แยกต่างหาก (requirement ต้นฉบับอ้างถึงแต่ระบบนี้มีแค่ admin/sale/finance — admin เห็น/จัดการได้ทั้งหมดอยู่แล้วจึงครอบคลุมส่วน "ดูสถานะรวม/รายงาน" ของ Manager ไปในตัว); ตัวกรอง "วันที่ขาย" (อิง `quotations.quot_date`) ยังไม่รองรับกรองฝั่ง server เพราะ join ผ่าน embedded resource ของ Supabase กรองแค่ข้อมูลที่ join มา ไม่กรองแถวหลัก — ใช้ตัวกรอง "วันที่ส่งคำขอ" (`created_at`) แทนไปก่อน; เกณฑ์ "งานเกินกำหนด" ในการ์ดสรุปใช้ค่าคงที่ 2 วันแบบง่ายๆ ไม่ได้ผูก SLA ต่อระดับความเร่งด่วน ปรับได้ที่ `fetchAccountingDocSummary()` ใน `api.js`
### ออเดอร์ (รันเลขออเดอร์เพื่อเปิดบิลในระบบบัญชีอื่น)
- เมนูใหม่ **"ออเดอร์"** ในกลุ่ม "การขาย" — ใช้รันเลขออเดอร์จากใบเสนอราคา เอาไปกรอกเปิดบิลต่อในระบบบัญชี/ERP ภายนอกอีกระบบหนึ่ง (ไม่ใช่ระบบเดียวกับ Payment Verification/Order Created ของโมดูลตรวจยอดโอน — นี่คือเลขอ้างอิงคนละชุดกัน)
- **เลขที่ออเดอร์** รูปแบบ `WTE{ปี 2 หลัก}WT{เลขรัน 4 หลัก}` เช่น `WTE26WT0001` — รันแยกตามปี ขึ้นปีใหม่เริ่มนับ 0001 ใหม่ (ตาราง `order_no_counters` + ฟังก์ชัน `gen_order_no()`) **รันทันทีตอนเปิดฟอร์มสร้างออเดอร์** ให้เห็นเลขล่วงหน้าได้เลย (ถ้าเปิดฟอร์มแล้วปิดไม่บันทึก เลขนั้นจะถูกข้ามไปเฉยๆ ไม่เอามาใช้ซ้ำ)
- **ขั้นตอนสร้างออเดอร์:** เลือก **เลขที่ใบเสนอราคา** (แสดงเฉพาะใบที่ยังไม่ถูกใช้เปิดออเดอร์อื่นอยู่) → ระบบคัดลอก **ข้อมูลบริษัท (ชื่อ/เลขประจำตัวผู้เสียภาษี/ที่อยู่/เบอร์โทร/อีเมล) + รายการสินค้าทั้งหมด** จากใบเสนอราคานั้นมาให้อัตโนมัติ (แก้ไขได้ก่อนบันทึก) — ข้อมูลบริษัทนี้ snapshot ไว้ในออเดอร์เอง ให้บัญชีเอาไปออกใบแจ้งหนี้/ใบกำกับภาษีทีหลังได้โดยไม่ต้องย้อนไปเปิดใบเสนอราคา → กรอก **ที่อยู่จัดส่ง** (บังคับ) + ผู้รับ/เบอร์โทร + หมายเหตุ (ไม่บังคับ) → ชื่อเซลล์ผู้เปิดออเดอร์ยึดตามบัญชีที่ login อยู่เสมอ (ช่องแสดงผลอย่างเดียว แก้ไม่ได้)
- กดปุ่ม **"บันทึกออเดอร์"** จะขึ้นป็อปอัปยืนยันก่อนเสมอ ระบุชัดว่าหลังบันทึกแล้วจะแก้ไขไม่ได้ ต้องยกเลิกแล้วเปิดออเดอร์ใหม่เท่านั้น
- **กฎสำคัญ:** ใบเสนอราคาหนึ่งใบผูกออเดอร์ที่ **"ใช้งานอยู่"** ได้แค่ 1 ออเดอร์พร้อมกัน (unique index ระดับ DB กันชนกันจริงๆ ไม่ใช่แค่เช็คฝั่ง frontend) — ออเดอร์ที่บันทึกแล้ว **แก้ไขไม่ได้อีก** ต้อง **"ยกเลิก"** เท่านั้น (พร้อมระบุเหตุผล) ถึงจะใช้เลขที่ใบเสนอราคาเดิมเปิดออเดอร์ใหม่ได้ — แต่ต้องรันเลขออเดอร์ใหม่เสมอ ใช้เลขเดิมต่อไม่ได้ กฎ "แก้ไขไม่ได้/ยกเลิกแล้วห้ามแก้ต่อ" นี้บังคับด้วย trigger ระดับ DB (`guard_orders_immutable`) ไม่ใช่แค่ซ่อนปุ่มฝั่งแอป
- **หมายเหตุการปรับให้เข้ากับระบบเดิม:** ระบบนี้ยังไม่มีตาราง/หน้า "ออเดอร์" ที่ผูกกับใบแจ้งหนี้จริงมาก่อน โมดูลนี้จึงเป็นตารางใหม่แยกต่างหาก (`orders`/`order_items`) ไม่ได้ใช้ร่วมกับ `payment_requests.order_no` (ที่กรอกเป็นข้อความอิสระตอน "เปิดออเดอร์" ในโมดูลตรวจยอดโอน) หรือ `accounting_document_requests.order_id` (ที่ผูกกับ `quotations` โดยตรง) — ทั้งสามจุดนี้เป็นแนวคิด "ออเดอร์" คนละบริบทกัน ในอนาคตถ้าต้องการรวมเป็นแหล่งเดียวกันจะต้องออกแบบ migration เพิ่ม
- ตาราง/ฟังก์ชันใหม่: `orders`, `order_items`, `order_no_counters`, `gen_order_no()`, `guard_orders_immutable()` — **ต้องรัน SQL migration ก่อนใช้งาน** (ดู `supabase/schema.sql`)
- **สิทธิ์:** ทุกคนที่ login เห็นออเดอร์ได้ทุกใบ (เหมือนโมดูลอื่น) — ยกเลิกออเดอร์ได้เฉพาะเจ้าของ (คอลัมน์ `sales_id`) หรือ admin เท่านั้น ไม่มี policy ลบถาวรเลย (ห้ามลบ ยกเลิกได้อย่างเดียว)

- **แจ้งเตือนฝ่ายบัญชีตอน Submit** (เมื่อ Sale กด "ส่งให้บัญชี" เรียก Netlify Function `notify-finance` ตัวเดียว ทำทั้ง 2 ช่องทางในคราวเดียว):
  1. **แจ้งเตือนในระบบ (กระดิ่งมุมบนขวา)** — ใช้งานได้ทันทีไม่ต้องตั้งค่าเพิ่ม insert แถวเข้าตาราง `notifications` ให้ทุกคนที่มีสิทธิ์ finance กดกระดิ่งจะเห็นรายการ ยังไม่อ่านมี badge ตัวเลขสีแดง กดรายการเพื่อพาไปหน้า "ตรวจสอบยอดโอน" และมาร์คอ่าน กระดิ่งโพลนับที่ยังไม่อ่านทุก 30 วินาที — ถ้ามีแจ้งเตือนใหม่เข้ามาระหว่างเปิดแอปทิ้งไว้ จะมีเสียง "ติ๊ง" สั้นๆ เตือนด้วย (สังเคราะห์เสียงด้วย Web Audio API ไม่มีไฟล์เสียงแนบ — เบราว์เซอร์บางตัวอาจต้องมี interaction ในหน้าเว็บก่อนถึงจะเล่นเสียงได้ตามนโยบาย autoplay)
  2. **อีเมล (ไม่บังคับ)** — ผ่าน **Resend**: ตั้ง `RESEND_API_KEY` (จาก resend.com) และ `NOTIFY_FROM_EMAIL` (อีเมลผู้ส่งที่ verify โดเมนกับ Resend แล้ว) ถ้าไม่ตั้งค่าระบบจะข้ามช่องนี้เงียบๆ

  ช่องทางไหนไม่ตั้งค่าไว้ ระบบจะข้ามช่องทางนั้นเงียบๆ ไม่กระทบการ Submit — อย่างน้อยช่องทางที่ 1 (in-app) ทำงานได้เสมอ
- ตาราง `notifications` (คอลัมน์ `user_id`/`title`/`body`/`entity_type`/`entity_id`/`link_view`/`read_at`) — insert ทำผ่าน Netlify Function ด้วย Service Role Key เท่านั้น (Sale ไม่มีสิทธิ์เห็น/เขียนแถวของผู้ใช้อื่นตาม RLS ปกติ) ส่วน select/mark-read จำกัดเฉพาะเจ้าของแถว — component `src/components/NotificationBell.jsx`
