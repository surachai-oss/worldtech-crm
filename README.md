# Worldtech B2B CRM (React + Supabase)

เวอร์ชัน rebuild จาก Google Apps Script เดิม ให้ deploy บน Netlify ได้

## โครงสร้าง
- **Frontend:** React + Vite (ธีมสี Navy/Yellow เดิม)
- **Backend/Database:** Supabase (Postgres + Auth)
- **PDF ใบเสนอราคา:** สร้างผ่าน browser print dialog (บันทึกเป็น PDF ได้ทันที ไม่ต้องพึ่ง Google Docs)

## ขั้นตอนติดตั้ง

### 1. สร้าง Supabase Project
1. ไปที่ https://supabase.com สร้างโปรเจกต์ใหม่ (ฟรีได้)
2. เปิด **SQL Editor** → New query → คัดลอกทั้งหมดจาก `supabase/schema.sql` มาวาง → Run
   - ไฟล์นี้จะสร้างตารางทั้งหมด, Storage bucket `attachments` สำหรับเอกสารแนบ, และระบบสิทธิ์ Admin/Sale ให้อัตโนมัติ
   - ไฟล์รันซ้ำได้เสมอ (idempotent) ถ้าอัปเดตโปรเจกต์ในอนาคตให้รันไฟล์นี้ทับได้เลย
3. ไปที่ **Authentication > Users** → เพิ่มผู้ใช้งาน (อีเมล/รหัสผ่าน) สำหรับพนักงานที่จะเข้าระบบ
   - หรือเปิด **Authentication > Providers** ตั้งค่า sign-up ถ้าต้องการให้สมัครเองได้
   - ทุกบัญชีใหม่จะได้สิทธิ์ **"พนักงานขาย" (sale)** โดยอัตโนมัติ
4. ไปที่ **Project Settings > API** คัดลอกค่า:
   - `Project URL` → ใช้เป็น `VITE_SUPABASE_URL`
   - `anon public` key → ใช้เป็น `VITE_SUPABASE_ANON_KEY`
5. ตั้งให้ตัวเองเป็น **Admin คนแรก** — กลับไปที่ SQL Editor แล้วรัน:
   ```sql
   update profiles set role = 'admin' where email = 'you@company.com';
   ```
   (ต้อง login เข้าระบบอย่างน้อย 1 ครั้งก่อน เพื่อให้มีแถวใน `profiles` ให้อัปเดต) หลังจากนั้นเมนู "ผู้ใช้งาน" จะปรากฏใน Sidebar ให้จัดการสิทธิ์คนอื่นต่อได้จากในแอปเลย

### 2. ตั้งค่าโปรเจกต์ในเครื่อง
```bash
npm install
cp .env.example .env
# แก้ .env ใส่ VITE_SUPABASE_URL และ VITE_SUPABASE_ANON_KEY ของคุณ
npm run dev
```
เปิด http://localhost:5173 แล้ว login ด้วยบัญชีที่สร้างไว้ในขั้นตอนที่ 1.3

### 3. ตั้งค่าข้อมูลบริษัท (สำหรับหัวใบเสนอราคา PDF) — ไม่บังคับ
ใน Supabase SQL Editor รัน:
```sql
insert into settings (key, value) values
  ('COMPANY_NAME', 'บริษัทของคุณ จำกัด'),
  ('COMPANY_ADDRESS', 'ที่อยู่บริษัท'),
  ('COMPANY_TAX_ID', '0-0000-00000-00-0'),
  ('COMPANY_PHONE', '02-xxx-xxxx'),
  ('COMPANY_EMAIL', 'contact@yourcompany.com'),
  ('COMPANY_LINE', '@yourlineid')
on conflict (key) do update set value = excluded.value;
```
`COMPANY_TAX_ID` และ `COMPANY_LINE` ไม่บังคับ — ถ้าไม่ตั้งค่าไว้ใบเสนอราคาจะไม่แสดงบรรทัดนั้น

## Deploy ขึ้น Netlify

### วิธีที่ 1: ผ่านเว็บ Netlify (ง่ายที่สุด)
1. Push โค้ดทั้งโฟลเดอร์นี้ขึ้น GitHub repo
2. ที่ Netlify → **Add new site > Import an existing project** → เลือก repo
3. Build command: `npm run build`, Publish directory: `dist` (ตั้งไว้ให้แล้วใน `netlify.toml`)
4. ไปที่ **Site settings > Environment variables** เพิ่ม:
   - `VITE_SUPABASE_URL`
   - `VITE_SUPABASE_ANON_KEY`
   - `SUPABASE_SERVICE_ROLE_KEY` — คัดลอกจาก Supabase **Project Settings > API > service_role** (⚠️ **ห้ามใส่ prefix `VITE_`** ไม่งั้น Vite จะฝังคีย์นี้ไปกับโค้ดฝั่ง browser ซึ่งเท่ากับเปิดสิทธิ์สูงสุดให้ใครก็ได้ที่เปิด devtools ดู — ต้องเป็นตัวแปรฝั่ง server สำหรับ Netlify Function เท่านั้น) ใช้สำหรับฟีเจอร์ admin เพิ่มผู้ใช้งานจากในแอป และฟอร์มลีดสาธารณะ (ดูหัวข้อ "ลีดจากฟอร์มสาธารณะ" ด้านล่าง) — ตัวแปรเดียวกัน ไม่ต้องตั้งเพิ่ม
5. Deploy

### วิธีที่ 2: Netlify CLI
```bash
npm install -g netlify-cli
netlify login
netlify init
netlify env:set VITE_SUPABASE_URL "https://xxxx.supabase.co"
netlify env:set VITE_SUPABASE_ANON_KEY "your-anon-key"
netlify env:set SUPABASE_SERVICE_ROLE_KEY "your-service-role-key"
netlify deploy --prod
```

### ทดสอบฟีเจอร์ "เพิ่มผู้ใช้งาน" ในเครื่อง
`npm run dev` (Vite) ไม่รัน Netlify Function ให้ ต้องใช้ Netlify CLI แทน:
```bash
npm install -g netlify-cli
netlify dev
```
แล้วตั้งค่า `SUPABASE_SERVICE_ROLE_KEY` ไว้ในไฟล์ `.env` ของเครื่อง (Netlify CLI จะอ่านจาก `.env` ให้ฟังก์ชันด้วย) หรือรันแล้ว deploy ขึ้น Netlify จริงเพื่อทดสอบก็ได้เช่นกัน

## ย้ายข้อมูลเดิมจาก Google Sheets
ถ้ามีข้อมูลอยู่ใน Google Sheet เดิมอยู่แล้ว:
1. Export แต่ละชีทเป็น CSV (File > Download > CSV)
2. ใน Supabase → Table Editor → เลือกตารางที่ตรงกัน (เช่น `companies`) → **Insert > Import data from CSV**
3. แม็ปคอลัมน์ให้ตรงกับชื่อฟิลด์ภาษาอังกฤษในตาราง (ดูชื่อคอลัมน์ได้จาก `supabase/schema.sql`)
4. ทำทีละตารางตามลำดับ: companies → contacts/deals → activities/tasks/quotations (เพราะมี foreign key อ้างถึงกัน)

## ฟีเจอร์ที่เพิ่มเข้ามา

### เอกสารแนบ (Attachments)
- แต่ละบริษัทมีแท็บ "เอกสารแนบ" ให้อัปโหลด/ดาวน์โหลด/ลบไฟล์ได้ (จำกัดไฟล์ละไม่เกิน 20MB)
- ไฟล์เก็บใน Supabase Storage bucket ชื่อ `attachments` (private bucket, เข้าถึงผ่าน signed URL อายุ 60 วินาทีเท่านั้น)

### สิทธิ์การใช้งาน Admin / Sale
- ผู้ใช้งานใหม่ทุกคนเริ่มต้นเป็น **พนักงานขาย (sale)** อัตโนมัติ — เห็น/แก้ไข/ลบได้เฉพาะข้อมูล (บริษัท, ดีล, งาน) ที่ตัวเองสร้าง บวกกับข้อมูลเก่าที่ยังไม่มีเจ้าของ
- **ผู้ดูแลระบบ (admin)** เห็น/แก้ไข/ลบได้ทุกอย่าง และมีเมนู "ผู้ใช้งาน" ในการปรับสิทธิ์คนอื่น
- ผู้ติดต่อ/กิจกรรม/ใบเสนอราคา/เอกสารแนบ สืบสิทธิ์ตามบริษัทแม่ (ถ้าเห็นบริษัทได้ ก็จัดการของในบริษัทนั้นได้)
- สิทธิ์ถูกบังคับที่ระดับฐานข้อมูล (Row Level Security) เสมอ — ปุ่มที่ซ่อนในหน้าเว็บเป็นแค่ UX เท่านั้น
- Admin เพิ่มผู้ใช้งานใหม่จากในแอปได้เลยที่หน้า "ผู้ใช้งาน" → **+ เพิ่มผู้ใช้งาน** (กรอกอีเมล/ชื่อ/ตั้งรหัสผ่านให้เลย ยังไม่มีอีเมลแจ้งอัตโนมัติ ต้องคัดลอกรหัสผ่านไปแจ้งพนักงานเอง)
  - ฟีเจอร์นี้รันผ่าน Netlify Function ที่ใช้ **Service Role Key** (สิทธิ์สูงสุด ข้าม RLS ทั้งหมด) — คีย์นี้อยู่ฝั่ง server เท่านั้น ต้องตั้งค่าเพิ่มก่อนใช้งานได้ (ดูขั้นตอนด้านล่าง)

### Pagination
- หน้า บริษัทลูกค้า / ผู้ติดต่อ / ประวัติการติดต่อ / งาน Follow-up / ใบเสนอราคา โหลดข้อมูลทีละหน้า (20 รายการ) จากฐานข้อมูลโดยตรง แทนการโหลดทั้งหมดมาไว้ในเบราว์เซอร์
- Dashboard, ค้นหาส่วนกลาง, หน้ารายละเอียดบริษัท (แท็บย่อย) และ Kanban ดีล ยังใช้ข้อมูลชุดเต็มเหมือนเดิม เพราะต้องคำนวณสรุปข้ามทั้งระบบ — เหมาะกับข้อมูลระดับหลักพันรายการ ถ้าข้อมูลโตกว่านี้มากควรแยกไปคำนวณสรุปฝั่ง Postgres (view/RPC) แทน

### นำเข้าข้อมูลจากไฟล์ Excel (บริษัท / สินค้า)
- หน้า "บริษัทลูกค้า" และ "สินค้า" มีปุ่ม **"นำเข้าจากไฟล์"** แบบเดียวกัน — ดาวน์โหลด Template (.xlsx) กรอกข้อมูลใน Excel แล้วอัปโหลดไฟล์กลับมาที่นี่ได้เลย (ไม่ต้อง save เป็น .csv อีกต่อไป) ระบบจะพรีวิวแถวที่ถูก/ผิดก่อนยืนยันนำเข้าเสมอ
- ใช้ไลบรารี `exceljs` อ่าน/สร้างไฟล์ Excel (ไม่ใช้ไลบรารี `xlsx` ที่มีบน npm เพราะเวอร์ชันบน npm ค้างที่เวอร์ชันเก่าที่มีช่องโหว่ความปลอดภัยยังไม่มีแพตช์) โค้ดกลางอยู่ที่ `src/lib/importExport.js` (`readExcelRows`, `downloadExcelTemplate`, `exportRowsToExcel`) ใช้ร่วมกันทุกฟีเจอร์ import/export ในระบบ
- นำเข้าสินค้า: กันรหัสสินค้าซ้ำให้อัตโนมัติทั้งกรณีซ้ำกับที่มีอยู่แล้วในระบบ และซ้ำกันเองภายในไฟล์ที่อัปโหลด
- Template ไฟล์บริษัทมี **dropdown ตัวเลือกฝังอยู่ในไฟล์ Excel เลย** (ไม่ต้องพิมพ์เอง กันพิมพ์ผิด) — คอลัมน์ "ประเภทลูกค้า"/"อุตสาหกรรม"/"สถานะ"/"ที่มา" ดึงค่าปัจจุบันจาก Picklists มาเป็นตัวเลือก — ค่าตัวเลือกเก็บอยู่ในชีตที่ซ่อนไว้ชื่อ "Lists" ในไฟล์ (มาจากข้อจำกัดของ Excel ที่ใส่ตัวเลือกยาวๆ ลงในสูตรตรงๆไม่ได้ถ้าเกิน 255 ตัวอักษร)
- ทุกครั้งที่เพิ่มคอลัมน์ใหม่ที่เป็น dropdown ในฟอร์ม (เช่น `customer_type`) ต้องอัปเดต template ให้มีคอลัมน์นั้นพร้อม dropdown ด้วยเสมอ ไม่ให้ template ตกรุ่นตามหลังฟอร์มจริง

### เลือกบริษัท/สินค้าแบบพิมพ์ค้นหา (Searchable Dropdown)
- Dropdown เลือก**บริษัท**และเลือก**สินค้า** (ในฟอร์มดีล/ใบเสนอราคา/ผู้ติดต่อ/กิจกรรม/งาน) พิมพ์ชื่อหรือบางส่วนของชื่อเพื่อกรองตัวเลือกได้เลย ไม่ต้องเลื่อนหาในลิสต์ยาวๆ — component อยู่ที่ `src/components/SearchableSelect.jsx`
- ตั้งใจใช้เฉพาะ dropdown ที่ตัวเลือกอาจมีจำนวนมาก (บริษัท/สินค้า) เท่านั้น ส่วน Dropdown แบบ Picklist (อุตสาหกรรม, สถานะ ฯลฯ) ที่ตัวเลือกน้อยและแก้ไขได้เองยังใช้ dropdown ปกติเหมือนเดิม

### แนบเอกสารตอนสร้างบริษัท
- ฟอร์มเพิ่ม/แก้ไขบริษัทมีช่องแนบเอกสาร (ภพ20, หนังสือรับรองบริษัท ฯลฯ) ได้เลยตอนสร้าง/แก้ไขบริษัท — ไฟล์จะถูกเก็บในระบบเอกสารแนบเดียวกับแท็บ "เอกสารแนบ"

### รายการสินค้าในดีล + คำนวณ VAT อัตโนมัติ
- หน้า "ดีลการขาย" มีตัวกรองช่วงวันที่สร้างดีล (จาก-ถึง) ที่แถบด้านบน — ใช้ดูดีลที่สร้างขึ้นในช่วง/เดือนนั้นๆ
- เหนือ Kanban มีการ์ด **"ดีลที่ต้องติดตาม (Follow-up) แยกตามเดือน"** — สรุปดีลที่ยังไม่ปิด (ไม่ใช่ Closed Won/Lost) ที่มีวันที่ต้อง Follow up ไว้ จัดกลุ่มตามเดือน เดือนที่เลยกำหนดมาแล้วไฮไลต์สีแดงให้เห็นชัด กันดีลตกหล่น (การ์ดนี้ไม่ขึ้นกับตัวกรองช่วงวันที่สร้างด้านบน — ดูภาพรวมทั้งหมดเสมอ ไม่แสดงถ้าไม่มีดีลที่ต้องติดตาม)
- ฟอร์มเพิ่ม/แก้ไขดีลมีเพิ่ม: **ที่มาของดีล** (ไลน์/เทเลเซลล์/อีเมลล์ ฯลฯ — แก้ไขตัวเลือกได้เฉพาะ admin เหมือน dropdown อื่น), **วันที่ต้อง Follow up**, และ **รายการสินค้า** ที่กดเพิ่มได้หลายรายการ (เลือกสินค้า + จำนวน + ราคาต่อหน่วย)
- ราคาต่อหน่วยที่กรอกถือว่า **รวม VAT แล้ว** — ระบบคำนวณยอดไม่รวม VAT, ยอด VAT (7%), และยอดรวมทั้งสิ้นให้อัตโนมัติจากผลรวมทุกรายการ
- **"มูลค่า" ของดีล (ที่ใช้คำนวณยอดใน Dashboard/Kanban) ถูกคำนวณจากรายการสินค้าโดยอัตโนมัติเสมอ ไม่มีช่องให้กรอกมูลค่าเองแล้ว** — ถ้าไม่เลือกรายการสินค้าเลย มูลค่าจะเป็น 0
- เมนู "สินค้า" (อยู่ในหมวด "การขาย" เห็นได้ทุกคนที่ login) ใช้จัดการรายชื่อสินค้าที่เลือกได้ในดีล/ใบเสนอราคา — มี **รหัสสินค้า + ชื่อสินค้า + รูปสินค้า (ไม่บังคับ)** ไม่ได้เก็บราคาไว้ล่วงหน้า เพราะราคาต่อหน่วยกรอกใหม่ทุกครั้งตอนสร้างดีล/ใบเสนอราคา
- **ทุกคนที่ login แล้วเพิ่ม/แก้ไข/ลบสินค้าได้หมด** ไม่ได้จำกัดแค่ admin (ต่างจาก Picklists ที่จำกัดแค่ admin) — ตามที่ตกลงกันไว้ ถ้าอยากจำกัดสิทธิ์แจ้งได้ จะแก้ RLS policy `products write` ให้
- รูปสินค้าเก็บใน Storage bucket **"product-images"** ซึ่งเป็น **public bucket** (ต่างจาก "attachments" ที่ private) เพราะรูปสินค้าไม่ใช่ข้อมูลลับ ทำให้ใช้ URL ตรงได้เลยไม่ต้องขอ signed URL — อัปโหลด/ลบรูปได้ทุกคนที่ login เหมือนกัน, ดูได้ทุกคน (แม้ยังไม่ login ก็ได้ เพราะ bucket เป็น public)

### สรุปใบเสนอราคาแยกตามสถานะ + ตัวกรองช่วงวันที่ + แนบไฟล์ที่ลูกค้าเซ็นกลับ
- หน้า "ใบเสนอราคา" มีการ์ดสรุปจำนวน+มูลค่าแยกตามสถานะ (Draft/Sent/Approved/Rejected/Expired) อยู่เหนือตาราง — อัปเดตตามตัวกรองค้นหา/ช่วงวันที่ที่ตั้งไว้เสมอ แต่ไม่ขึ้นกับตัวกรอง "สถานะ" ตัวเดียว (ตั้งใจให้เห็นทุกสถานะพร้อมกัน)
- มีตัวกรองช่วงวันที่ใบเสนอราคา (จาก-ถึง) อิงจาก `quot_date`
- ทั้งหน้ารายการใบเสนอราคาและแท็บ "ใบเสนอราคา" ในหน้ารายละเอียดบริษัท มีปุ่ม **"แนบไฟล์เซ็นกลับ"** สำหรับอัปโหลดใบเสนอราคาที่ลูกค้าเซ็นแล้วส่งกลับมา (เก็บใน bucket "attachments" เดียวกับเอกสารแนบอื่น) — เมื่อแนบแล้วปุ่มจะเปลี่ยนเป็น "ไฟล์เซ็นกลับ" (ดาวน์โหลด) และ "ลบไฟล์เซ็นกลับ"
- ฟอร์มเพิ่ม/แก้ไขบริษัทเพิ่มช่อง **"เลขประจำตัวผู้เสียภาษี"** ใช้แสดงในใบเสนอราคาที่พิมพ์
- ใบเสนอราคาแก้ไขได้แล้ว — ปุ่ม **"แก้ไข"** เปิดฟอร์มเดิมพร้อมข้อมูลเดิมให้แก้ก่อนพิมพ์ ไม่ต้องลบทิ้งแล้วสร้างใหม่ (ทั้งหน้ารายการและแท็บในหน้าบริษัท)
- ฟอร์มใบเสนอราคารองรับ **หลายรายการสินค้าต่อใบ** เหมือนดีล — กด "+ เพิ่มรายการ" เพิ่มแถวได้ไม่จำกัด แต่ละแถวเลือกสินค้าได้ (ไม่บังคับ, เติมชื่อรายการ+โชว์รูปสินค้าอัตโนมัติ) หรือพิมพ์ชื่อรายการเองก็ได้ (เผื่อของที่ไม่มีในรายการสินค้า) พร้อมจำนวน+ราคาต่อหน่วย (ถือว่ารวม VAT แล้ว) — **"มูลค่ารวม" คำนวณจากผลรวมทุกรายการอัตโนมัติ ไม่มีช่องกรอกมูลค่าเองแล้ว** เหมือนกับดีล
- มีช่อง **"เบอร์ติดต่อเซลล์"** ต่อใบเสนอราคา ใช้แสดงในกล่องข้อมูลติดต่อตอนพิมพ์
- มีช่อง **"ชื่อผู้เสนอราคา"** (เติมชื่อผู้ใช้งานที่ล็อกอินให้อัตโนมัติตอนสร้างใบใหม่ แก้ไขได้) พิมพ์แสดงอยู่เหนือเส้น "ผู้เสนอราคา" ในเอกสารที่พิมพ์ออกมาเลย ไม่ต้องพรินซ์ออกมาเซ็นสดก่อนส่งลูกค้า
- ช่อง **"หมายเหตุ"** เติมเงื่อนไขมาตรฐานของบริษัท (การจัดส่ง/รับประกัน ฯลฯ) ให้อัตโนมัติทุกครั้งที่สร้างใบเสนอราคาใหม่ — แก้ไข/ลบข้อความได้ตามเงื่อนไขที่ตกลงกับลูกค้าจริง (เทมเพลทอยู่ใน `src/components/Modals.jsx` ค่าคงที่ `DEFAULT_QUOTATION_NOTE` แก้ข้อความเทมเพลทได้ที่นี่)
- ใบเสนอราคาเก่าก่อนมีระบบรายการสินค้า (มีแค่ subject/value เดี่ยว) จะถูกย้ายเข้าระบบรายการสินค้าให้อัตโนมัติตอนรัน schema.sql (ถ้าตอนนั้นมีการเลือกสินค้าไว้แล้ว) หรือแปลงเป็นรายการเดียวให้อัตโนมัติตอนเปิดแก้ไขครั้งแรก (ถ้าเป็นใบที่กรอกแบบเดิมไม่มีสินค้า) — ข้อมูลเดิมไม่หายแน่นอน

### รูปแบบใบเสนอราคา PDF
- ปรับรูปแบบใบเสนอราคาที่พิมพ์/บันทึกเป็น PDF ใหม่ทั้งหมด ให้ตรงกับแบบฟอร์มที่บริษัทใช้จริง — แถบหัวกระดาษสีกรมท่า (มีคำว่า QUOTATION ภาษาอังกฤษด้วย) โลโก้ + ข้อมูลบริษัท, เลขที่/วันที่, ข้อมูลลูกค้าพร้อมเลขผู้เสียภาษี, ตารางรายการสินค้าหลายแถว (พร้อมรูปสินค้าถ้ามี), กล่องสรุปยอด (ไม่รวม VAT / VAT 7% / ยอดชำระ), หมายเหตุ, และกล่องข้อมูลติดต่อ (รวมเบอร์เซลล์ถ้ากรอกไว้)
- ตัวเลขในใบเสนอราคาถือว่า **ราคาต่อหน่วยที่กรอกรวม VAT แล้ว** เหมือนกับดีล — ระบบคำนวณยอดไม่รวม VAT และ VAT 7% ย้อนกลับให้อัตโนมัติจากผลรวมทุกรายการ
- โลโก้ที่ใช้อยู่ที่ `public/worldtech-logo.png`
- `COMPANY_ADDRESS` ใส่ตัวอักษรขึ้นบรรทัดใหม่ (newline) ในค่าที่บันทึกได้ เพื่อคุมให้ที่อยู่ตัดบรรทัดสั้นๆ ตามที่ต้องการเอง — ระบบไม่ตัดคำให้อัตโนมัติแล้ว (กันตัด `\n` ที่ตั้งใจไว้ผิดที่) ฝั่งซ้าย (โลโก้+ที่อยู่) และฝั่งขวา (วันที่/เลขที่) แบ่งพื้นที่กันเอง ไม่ชนกัน
- กดพิมพ์แล้วหน้าต่างจะเปิดทันที (กันเบราว์เซอร์บล็อกป๊อปอัป) แล้วค่อยโหลดรายการสินค้า/รูปมาแสดงทีหลัง — ถ้าเห็น "กำลังโหลดข้อมูล..." ค้างนานผิดปกติให้เช็คอินเทอร์เน็ต/สิทธิ์เข้าถึงข้อมูล

### เชื่อมข้อมูลระหว่างดีลกับใบเสนอราคา (ไม่ต้องกรอกซ้ำสองที่)
- ปุ่ม **"ออกใบเสนอราคา"** ที่การ์ดดีล (หน้า Kanban ดีลการขาย และแท็บ "ดีล" ในหน้าบริษัท) — เปิดฟอร์มใบเสนอราคาใหม่พร้อมก็อปบริษัท/หัวข้อ/รายการสินค้าทั้งหมดจากดีลมาให้ทันที ไม่ต้องพิมพ์ซ้ำ กดได้ไม่จำกัดจำนวนครั้ง (ดีลเดียวออกใบเสนอราคาได้หลายใบ)
- ปุ่ม **"สร้างดีล"** ที่แถวใบเสนอราคา (หน้า "ใบเสนอราคา" และแท็บ "ใบเสนอราคา" ในหน้าบริษัท) — เปิดฟอร์มดีลใหม่พร้อมก็อปบริษัท/รายการสินค้าจากใบเสนอราคามาให้ ปุ่มนี้จะไม่แสดงถ้าใบเสนอราคานั้นผูกกับดีลอยู่แล้ว (กันสร้างดีลซ้ำจากใบเดียวกันโดยไม่ตั้งใจ)
- ทั้งสองฝั่งผูกกันด้วยคอลัมน์ `quotations.deal_id` (มีอยู่ในสคีมาเดิมแล้วแต่ไม่เคยถูกใช้งาน) — เมื่อผูกแล้วจะเห็น badge เล็กๆ ทั้งสองทาง เช่น ที่การ์ดดีลโชว์ "ออกใบเสนอราคาแล้ว N ใบ" และที่แถวใบเสนอราคาโชว์ "จากดีล: ชื่อดีล"
- **คัดลอกครั้งเดียว ไม่ sync ต่อเนื่อง** — กดปุ่มแล้วรายการสินค้าจะถูกก็อปเป็นชุดใหม่ไปยังอีกฝั่ง แก้ไขต่อได้อิสระทั้งสองฝั่งโดยไม่กระทบกัน (เช่น ดีลมีสินค้า A แต่ตอนออกใบเสนอราคาลูกค้าขอเพิ่มสินค้า B ก็เพิ่มแถวในฟอร์มใบเสนอราคาต่อได้เลย ดีลเดิมไม่ถูกแก้ตาม)
- โครงสร้างรายการสินค้าของดีล (`deal_items`) และใบเสนอราคา (`quotation_items`) เหมือนกันเป๊ะ (`product_id` + `description` + `quantity` + `unit_price`) ก็อปไปมาได้ตรงๆ ไม่มีข้อจำกัด รวมถึงแถวที่พิมพ์ชื่อรายการเอง ไม่ได้เลือกจากรายการสินค้าในระบบ ก็ก็อปได้ปกติ

### ลูกค้าบุคคลธรรมดา / นิติบุคคล
- ฟอร์มเพิ่ม/แก้ไข **"ลูกค้า"** (ยังใช้ตาราง `companies` เดียวกันเหมือนเดิม ไม่ได้แยกตารางใหม่ — เลือกได้จาก dropdown เดียวกันตอนสร้างดีล/ใบเสนอราคาเหมือนกันหมด) มีช่อง **"ประเภทลูกค้า"** ให้เลือก **นิติบุคคล/บริษัท** (ฟอร์มเต็มแบบเดิม) หรือ **บุคคลธรรมดา** (ฟอร์มย่อ — ซ่อนช่ออุตสาหกรรม/เว็บไซต์/เลขประจำตัวผู้เสียภาษีที่ไม่เกี่ยวกับบุคคล)
- เหมาะกับลูกค้าที่ซื้อสินค้าไปใช้ส่วนตัว ไม่มีข้อมูลบริษัท แต่ยังต้องการใบเสนอราคา — กรอกแค่ชื่อ-เบอร์-ที่อยู่สั้นๆ แล้วเลือกได้จาก dropdown ตอนสร้างดีล/ใบเสนอราคาเหมือนลูกค้าบริษัททุกอย่าง ไม่ต้องพิมพ์ข้อมูลซ้ำมือทุกครั้ง
- หน้า "บริษัทลูกค้า" มีตัวกรอง "ประเภทลูกค้า" เพิ่ม และมี badge เล็กๆ "บุคคล" กำกับไว้ที่ชื่อในรายการ ให้แยกจากลูกค้านิติบุคคลได้ง่าย
- คอลัมน์ "ประเภทลูกค้า" ในไฟล์ template นำเข้าบริษัทลูกค้าก็มี dropdown ตัวเลือกให้เหมือนคอลัมน์อื่นๆ (ดูหัวข้อ "นำเข้าข้อมูลจากไฟล์ Excel" ด้านบน)
- ฟอร์มเพิ่ม/แก้ไข **ดีล** และ **ใบเสนอราคา** มีปุ่ม **"+ ลูกค้าใหม่"** ข้าง dropdown เลือกบริษัท — กดแล้วเปิดฟอร์มลูกค้าใหม่ (เลือกประเภทบุคคล/นิติบุคคลได้เหมือนกัน แต่ไม่มีช่องแนบเอกสารในฟอร์มย่อนี้) ให้กรอก บันทึกแล้วลูกค้าใหม่จะถูกเลือกให้อัตโนมัติทันทีโดยไม่ต้องปิดฟอร์มดีล/ใบเสนอราคาที่กำลังกรอกอยู่ — กันปัญหาลูกค้าใหม่ (โดยเฉพาะนิติบุคคลที่ยังไม่มีอยู่ในระบบ) เลือกไม่ได้เพราะต้องออกไปสร้างที่หน้า "บริษัทลูกค้า" ก่อน

### ผู้ติดต่อ (ลีดจากฟอร์มสาธารณะ Facebook/เว็บไซต์) + แปลงเป็นลูกค้า
- เมนู "ผู้ติดต่อ" เดิม (รายชื่อคนติดต่อที่ผูกกับบริษัทลูกค้าโดยตรง) ถูกเอาออกจากหน้าหลักแล้ว — การจัดการผู้ติดต่อของแต่ละบริษัทยังทำได้ปกติผ่านแท็บ "ผู้ติดต่อ" ในหน้ารายละเอียดบริษัท ไม่ได้หายไปไหน แค่ไม่มีหน้ารายการกลางแยกอีกต่อไป
- เมนู **"ผู้ติดต่อ"** ในปัจจุบัน คือหน้ารวมลีดที่เข้ามาจากฟอร์มสาธารณะ (เดิมชื่อ "ลีดที่เข้ามา") — มีฟอร์มสาธารณะที่ **`/lead`** (เช่น `https://yourapp.netlify.app/lead?src=facebook`) ไม่ต้อง login เอาลิงก์ไปแปะใน Facebook Ads หรือเว็บไซต์ได้เลย — ลูกค้ากรอก **หัวข้อที่ต้องการติดต่อ**, ชื่อ-เบอร์โทร (บังคับทั้ง 3 ช่อง) และอีเมล/สินค้าที่สนใจ/ข้อความ (ไม่บังคับ) ที่มา (`source`) ถูกแท็กอัตโนมัติจาก query param `?src=` ในลิงก์ ไม่ต้องให้ลูกค้ากรอกเอง
- ข้อมูลที่กรอกจะถูกส่งผ่าน **Netlify Function `submit-lead`** (ใช้ Service Role Key เขียนเข้า Supabase แทน — ดูตัวแปร `SUPABASE_SERVICE_ROLE_KEY` ด้านบน) **ไม่ได้เปิด RLS ให้คนนอก (anon) insert เข้าตาราง `leads` ตรงๆ** เพื่อกันสแปม/ข้อมูลมั่วเขียนเข้าระบบได้ง่ายเกินไป
- หน้า "ผู้ติดต่อ" ให้เซลล์เห็นรายชื่อลีดที่กรอกเข้ามา (หัวข้อ/ชื่อ/เบอร์/สินค้าที่สนใจ/ที่มา) พร้อมตัวกรองสถานะ/ค้นหา, ปุ่ม **"ส่งออกเป็น Excel"** (ส่งออกตามตัวกรองที่ตั้งไว้ ไม่จำกัดแค่หน้าที่กำลังดูอยู่ เผื่อเอาไปใช้ต่อกับระบบอื่น) และปุ่ม **"สร้างเป็นลูกค้า"** — กดแล้วก็อปหัวข้อ/ชื่อ/เบอร์/อีเมล/ข้อความไปตั้งต้นฟอร์มลูกค้าใหม่ให้ (ตั้งประเภทเป็น **บุคคลธรรมดา** โดย default เพราะลีดส่วนใหญ่คือคนซื้อไปใช้เอง แก้เป็นนิติบุคคลได้ถ้าจริงๆเป็นบริษัท) แก้ไข/เติมข้อมูลที่ลีดกรอกมาไม่ครบได้ก่อนบันทึกจริง
- เมื่อแปลงเป็นลูกค้าแล้ว ปุ่ม "สร้างเป็นลูกค้า" จะเปลี่ยนเป็นปุ่มลิงก์ไปหน้าลูกค้าที่แปลงมาแล้วแทน (กันกดสร้างซ้ำจากลีดเดียวกัน) — เหมือน pattern การผูก deal↔quotation ด้านบน คือก็อปครั้งเดียวแล้วแก้ไขอิสระต่อได้ ไม่ sync ต่อเนื่อง
- ลบลีดได้เฉพาะ admin (สถานะ/แปลงเป็นลูกค้าทำได้ทุกคนที่ login เพราะลีดไม่มีเจ้าของเฉพาะคน เซลล์คนไหนก็ตามต่อได้)

### Dropdown แก้ไขได้เอง (Picklists)
- Dropdown ทุกตัวที่ใช้กรอกข้อมูล (อุตสาหกรรม, สถานะบริษัท, Stage ดีล, ประเภทกิจกรรม, ลำดับความสำคัญ/สถานะงาน, สถานะใบเสนอราคา, ที่มาลูกค้า, ที่มาของดีล) มีปุ่ม **"แก้ไข"** ข้างๆ ให้เพิ่ม/ลบตัวเลือกได้ทันทีโดยไม่ต้องไปหน้าตั้งค่าแยก — คล้าย dropdown list ที่แก้ไขได้ใน Google Sheets
- **เฉพาะผู้ดูแลระบบ (admin)** เท่านั้นที่เห็นปุ่ม "แก้ไข" และแก้ไข/เพิ่ม/ลบตัวเลือกได้ — พนักงานขายเลือกจากตัวเลือกที่มีอยู่ได้อย่างเดียว บังคับจริงที่ RLS policy `picklists write` (ทุกคน login แล้วยังอ่าน/เลือกตัวเลือกได้ปกติผ่าน `picklists select`)
- ⚠️ **ข้อควรระวัง:** ค่าบางตัวถูกผูกกับ logic ในโค้ดตรงๆ (`Closed Won`/`Closed Lost` ใช้คำนวณยอดดีลที่ปิดสำเร็จ, `เสร็จสิ้น` ใช้เช็คงานที่เสร็จแล้ว, `Active` ใช้นับบริษัท Active ใน Dashboard) ถ้าลบหรือเปลี่ยนชื่อค่าพวกนี้ ตัวเลขสรุปที่เกี่ยวข้องจะไม่ถูกต้องจนกว่าจะรีแฟกเตอร์โค้ดส่วนนั้นเพิ่ม (ตามที่ตกลงกันว่ายอมรับความเสี่ยงนี้ไว้ก่อน)
- รายการเริ่มต้นทั้งหมดถูก seed มาจากค่าเดิมที่เคยฮาร์ดโค้ดไว้ในโค้ด (ไม่มีอะไรหายไป) เก็บอยู่ในตาราง `picklists` ใหม่ — หน้า "ที่มาลูกค้า" เดิมถูกลบออกแล้ว เพราะปุ่ม "แก้ไข" ทำหน้าที่แทนได้ในตัว
